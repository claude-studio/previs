# `/previs-plan`

합의된 구현 계획을 리뷰어가 승인할 수 있는 previs plan 문서로 발행한다. 입력은
호출 문맥의 TRIP plan 문서, 대화 중 합의된 계획, 또는 사용자가 전달한 계획이다.
계획이 충분히 구체적이지 않으면 발행하기 전에 필요한 내용을 확인한다.

## 문서 작성

다음 순서로 JSON 블록 배열을 구성한다. 계획에 해당하지 않는 블록은 생략한다.

1. `wireframe` 또는 `diagram` — 계획의 구조와 검토 흐름을 보여주는 헤드라인
2. `prose` — 왜 이 작업이 필요한지와 어떤 접근을 취하는지 1~3문단
3. `file-tree` — 실제 작업 대상 파일과 변경 목적
4. `question-form` — 구현 전에 결정해야 할 미해결 질문
5. `callout` — 리뷰어에게 승인 또는 피드백을 요청하는 게이트

콘텐츠를 작성할 때 다음 규칙을 지킨다.

- 구조화 블록에는 계획에서 확인할 수 있는 파일·변경·관계만 기록한다. 추론은
  `prose`에 두고, 확정되지 않은 내용은 `question-form`으로 분리한다.
- 와이어프레임은 시맨틱 HTML 조각과 `--wf-*` 토큰을 사용한다. hex 색상,
  `font-family`, 고정 폭·높이·좌표를 하드코딩하지 않는다.
- 다이어그램은 계획에 실제로 필요한 관계만 표현한다.
- 제목은 약 70자 이하로 유지하고, API 키·토큰·`.env` 값은 `<redacted>`로
  마스킹한다.
- 문서의 `schemaVersion`은 `1`, `kind`는 `plan`으로 설정하고 모든 블록에
  고유한 `id`를 부여한다.

## 발행 및 검증

1. 제목에서 소문자 영숫자와 하이픈으로 된 짧은 slug를 만든다.
2. `id`를 `plan-YYYYMMDD-<slug>` 형식으로 만들고 `.previs/<id>.json`에
   저장한다. 날짜는 현재 날짜를 `YYYYMMDD`로 사용한다.
3. 같은 파일명이 이미 있으면 기존 파일을 읽거나 덮어쓰지 말고 `-2`, `-3`처럼
   사용 가능한 접미사를 붙여 id와 파일명을 함께 유일하게 만든다.
4. 저장 직후 다음 명령으로 검증한다.

   ```bash
   pnpm doc:validate .previs/<id>.json
   ```

5. 검증에 실패하면 JSON 블록을 수정하고 성공할 때까지 다시 검증한다. 검증 실패
   원본도 임의로 삭제하지 않아 수정할 수 있게 한다.

이번 스킬의 발행 대상은 previs 리포지토리 루트의 `.previs/`로 한정한다. 외부
대상 프로젝트나 `PREVIS_DOCS_DIR` 전달은 후속 런처 작업의 범위다.

## 뷰어 확인과 승인 게이트

뷰어 dev 서버가 이미 실행 중이면 출력된 로컬 URL을 열어 발행 문서를 확인하도록
안내한다. 서버가 실행 중이지 않으면 다음 명령으로 직접 기동하도록 안내한다.

```bash
pnpm --filter @previs/viewer dev
```

발행 후에는 리뷰어가 문서의 구조와 파일 맵을 확인하고 승인하거나 피드백을 줄
때까지 구현을 시작하지 않는다고 명시한다.
