# 로컬 발행 파이프라인 + /previs-plan 스킬 구현 계획 (M4 사이클 A)

## Overview

M4(에이전트 스킬)의 첫 사이클로, previs의 "생산" 계층을 연다. 에이전트가
계획을 JSON 블록 문서로 작성해 대상 프로젝트의 `.previs/` 폴더에 발행하는
`/previs-plan` 스킬과, 뷰어가 그 폴더를 로드하는 Vite 서버 API를 구현한다.
`/previs-recap`(diff 기계 도출)은 사이클 B(v0.5.0)에서 이 파이프라인 위에
구현한다.

## Problem Statement

- 뷰어는 내장 픽스처와 수동 파일 열기만 지원한다. "스킬 호출 → 수 초 내
  뷰어 확인"(GOAL §속도)을 만족하는 발행 경로가 없다.
- 에이전트가 previs 문서를 생성하는 표준 절차(스킬)가 없어, 블록 문서
  작성이 매번 즉흥적이다.
- 스킬 명칭이 문서상 `/plan`·`/recap`으로 표기돼 있으나 Claude Code 내장
  plan 모드·타 스킬과 충돌 여지가 있다 (사용자 결정: `/previs-plan`·
  `/previs-recap`으로 확정, 문서 표기 갱신 필요).

## Solution Architecture

```
/previs-plan 스킬 (skills/previs-plan/SKILL.md)
    │  블록 문서 작성 → pnpm doc:validate 검증
    ▼
.previs/<document-id>.json          ← 발행 규약 (git 추적)
    ▲
    │  GET /api/documents (raw pass-through)
Vite dev 서버 플러그인 (apps/viewer/server/)
    │
    ▼
뷰어: mount·window focus 시 fetch → safeParsePrevisDocument
     → origin 'published'로 목록 병합
```

- **문서 경계 유지**: 서버는 JSON을 검증 없이 전달만 하고, 클라이언트의
  `safeParsePrevisDocument`가 유일한 신뢰 경계로 남는다 (ARCHI §8).
- **스킬 소스 단일 원천**: 리포 루트 `skills/`가 제품 디렉토리(M7 패키징
  단위)이고, `.claude/skills/`에는 심링크로 연결해 이 리포에서 도그푸딩한다
  (CLAUDE.md → AGENTS.md 심링크 선례).
- **발행 규약**: 파일명 = 문서 id (`plan-YYYYMMDD-<slug>.json`). `.previs/`는
  git 추적한다 (ARCHI §9 "git으로 버전 관리", GOAL §데이터 주권).

## Implementation Details

### 1. Vite 서버 플러그인 — 발행 문서 서빙

**File**: `apps/viewer/server/previs-docs-plugin.ts` (new)

`configureServer` 훅으로 미들웨어를 등록하는 Vite 플러그인.

- `GET /api/documents`: 문서 디렉토리의 `*.json`(직속 파일만, 재귀 없음)을
  읽어 `{ documents: [{ fileName, mtimeMs, raw }], errors: [{ fileName,
message }] }`로 응답. JSON 파싱 실패 파일은 `errors`에 담고 나머지는
  정상 응답한다.
- 문서 디렉토리: `PREVIS_DOCS_DIR` 환경변수 우선, 기본값
  `<repoRoot>/.previs`. 디렉토리가 없으면 빈 목록 응답.
- 스키마 검증은 하지 않는다(raw pass-through) — 검증은 클라이언트 경계에서.
- 디렉토리 밖 경로 접근 경로 없음(목록 조회만, 파일명 파라미터 미수용).

**File**: `apps/viewer/vite.config.ts` (modify)

- 플러그인 import 및 `plugins` 배열 등록 (기존 포트/앨리어스 로직 유지).

### 2. 뷰어 문서 저장소 — published origin

**File**: `apps/viewer/src/lib/documents.ts` (modify)

- `DocumentOrigin`에 `'published'` 추가.
- `fetchPublishedDocuments()` 추가 — `/api/documents` fetch → 각 `raw`를
  기존 `parseDocument(raw, 'published')`로 검증. 반환은 판별 유니언:
  `{ ok: true; entries; errors } | { ok: false; message }`. 네트워크
  실패·HTTP 오류·페이로드 형식 위반은 `ok: false`로 구분한다 —
  "성공한 빈 디렉토리"와 "전송 실패"를 절대 동일 취급하지 않는다.
- 서버 응답 페이로드 타입은 플러그인과 공유한다 (§4).

### 3. 앱 상태 — published 문서 병합

**File**: `apps/viewer/src/App.tsx` (modify)

- mount 시 + `window` focus 시 `fetchPublishedDocuments()` 호출
  ("발행 → 브라우저 전환 → 즉시 반영" UX, 폴링 없음).
- 상태를 origin별로 분리 유지하고 렌더 시 병합. id 충돌 우선순위:
  `opened > published > builtin` (사용자가 명시적으로 연 파일이 자동
  갱신으로 덮이지 않도록).
- published 교체는 **성공 응답(`ok: true`)일 때만** 수행(삭제된 파일도
  반영). `ok: false`면 마지막 성공 항목을 유지하고 새로고침 오류를
  경고로 노출 — 서버가 잠시 죽었다고 문서 목록이 비어 보이면 안 된다.
- 새로고침 오류는 `fileError`와 **별도 상태**(`publishedError`)로
  관리한다 — 파일 열기 오류와 수명이 독립이라 한쪽 해제가 다른 쪽을
  가리거나 오래된 경고를 남기면 안 된다. `publishedError`는 다음 성공
  refetch에서 해제한다. 표시(경고 배너 스타일)만 기존 방식을 재사용.

### 3-1. 문서 뷰 — origin 라벨

**File**: `apps/viewer/src/components/app/DocumentView.tsx` (modify)

### 6. /previs-plan 스킬

**File**: `skills/previs-plan/SKILL.md` (new)

에이전트가 계획을 previs 블록 문서로 발행하는 절차서:

0. **frontmatter**: `name: previs-plan`, `description`,
   `disable-model-invocation: true` — 파일을 쓰는 워크플로우이므로 계획
   문맥만으로 모델이 자동 발행하지 못하게 명시 호출 전용으로 제한.
1. **입력**: 호출 문맥의 계획 내용(TRIP plan 문서, 대화 중 합의된 계획 등).
2. **plan 문서 골격** (프로젝트 개요의 plan 정의 반영): 헤드라인
   wireframe 또는 diagram → 서사 prose(왜·접근) → file-tree(작업 파일 맵)
   → question-form(미해결 질문) → callout(승인 요청). 해당 없는 블록은 생략.
3. **콘텐츠 규칙 준수**: 와이어프레임 `--wf-*` 체계(hex·좌표 하드코딩 금지),
   시크릿 마스킹, 타이틀 ~70자 (AGENTS.md 콘텐츠 규칙).
4. **발행**: id `plan-YYYYMMDD-<slug>` 부여, `.previs/<id>.json` 저장.
   **동일 파일명이 이미 존재하면 덮어쓰지 않고** `-2`, `-3` 접미사로 유일한
   id·파일명을 확보한다(같은 날 같은 slug의 조용한 덮어쓰기 방지).
   저장 후 `pnpm doc:validate`로 검증 — 실패 시 수정 후 재검증 루프.
5. **뷰어 안내**: dev 서버가 떠 있으면 문서 URL 안내, 없으면
   `pnpm --filter @previs/viewer dev` 수동 기동 안내 (M5 런처 전 임시).
6. **승인 게이트**: 리뷰어 확인·승인 후 구현 착수를 명시.

**이번 사이클 범위**: 발행 대상은 previs 리포 루트의 `.previs/`로
한정한다(도그푸딩). 외부 대상 프로젝트 발행과 `PREVIS_DOCS_DIR` 핸드오프
(런처가 대상 프로젝트 경로를 뷰어 서버에 전달)는 M5에서 설계한다.

**File**: `.claude/skills/previs-plan` (new, 심링크 → `../../skills/previs-plan`)

## Technical Considerations

- **Pattern Usage**: 문서 검증·저장소는 기존 `lib/documents.ts`의
  `parseDocument` 경로를 확장한다. 라우팅·목록 UI는 변경하지 않는다.
- **신뢰 경계**: 서버 플러그인은 파일 시스템 → HTTP 전달만 담당. 발행
  문서도 열린 파일과 동일하게 클라이언트 safeParse를 통과해야 목록에
  들어간다.
- **서버 부재 내성**: `vite build` 정적 산출물이나 API 미탑재 환경에서
  fetch 실패 시 마지막 성공 상태를 유지(최초 실패면 빈 목록)하고 오류를
  경고로 알리며, 뷰어의 나머지 기능은 그대로 동작한다.
- **스킬은 코드가 아니라 문서 자산**: TypeScript-only 규칙은 소스에만
  적용된다. 스킬 SKILL.md에는 실행 코드가 없고 절차와 규약만 담는다.
- **Edge Cases**: `.previs/` 부재(빈 목록), 깨진 JSON(errors로 보고),
  스키마 위반 문서(클라이언트 경고 노출), 전송 실패(마지막 성공 목록
  유지 + 오류 경고), 발행 문서와 열린 파일 id 충돌(opened 우선), 발행
  파일명 충돌(접미사로 유일화, 덮어쓰기 금지), focus refetch 중복 호출
  (in-flight 가드).

## Type Definitions

- `DocumentOrigin`: `'builtin' | 'opened'` → `'builtin' | 'opened' | 'published'`
- `PublishedDocumentsPayload`(new): `{ documents: { fileName: string;
mtimeMs: number; raw: unknown }[]; errors: { fileName: string;
message: string }[] }`
- `PublishedFetchResult`(new, 판별 유니언): `{ ok: true; entries:
DocumentEntry[]; errors: string[] } | { ok: false; message: string }`
- 스키마 패키지 공개 API 변경 없음 (CLI는 내부 엔트리).

## Performance & Cost Impact

- focus당 fetch 1회 + 로컬 디렉토리 읽기 — 무시 가능. 폴링·watch 없음.

## Backward Compatibility

- 스키마·기존 문서 포맷 변경 없음(additive). 기존 내장 픽스처·파일 열기
  동작 유지. 스킬 명칭은 아직 구현된 적 없어 rename 영향 없음.
