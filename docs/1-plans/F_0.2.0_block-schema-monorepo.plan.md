# 블록 스키마 & 모노레포 스캐폴딩 (M1) Implementation Plan

> 버전: 0.2.0 (M1 완료 시 minor 상승 — GOAL.md TRIP 운영 규칙)
> 범위: GOAL.md M1 전체 (사용자 확정: 단일 TRIP 사이클)

## Overview

previs의 모든 계층(에이전트 스킬·뷰어 렌더러·저장)이 공유하는 계약인
**블록 JSON 스키마**를 고정하고, 이를 담을 **pnpm workspace 모노레포**와
TypeScript·lint·test 툴체인을 확정한다. 산출물은 `packages/schema` 패키지
(12종 블록 타입 정의 + zod 런타임 검증 + 샘플 문서 픽스처)와 루트 툴체인
(`pnpm lint / typecheck / test` 동작)이다.

## Problem Statement

- 저장소에 코드가 전혀 없다. M2(뷰어)·M4(스킬)는 모두 블록 스키마를 계약으로
  삼으므로, 스키마가 먼저 고정되지 않으면 후속 마일스톤이 착수 불가능하다.
- ARCHI.md §3·§6·§14의 툴체인 항목이 `(예정)`/미정 상태다. TRIP 스킬이
  참조할 검증 명령(`pnpm lint/typecheck/test`)이 확정돼야 한다.

## Solution Architecture

```
previs/  (pnpm workspace 루트)
├── package.json            # private, 루트 스크립트: lint/typecheck/test/build
├── pnpm-workspace.yaml     # packages: ["packages/*", "apps/*"]
├── tsconfig.base.json      # strict 공통 옵션 (패키지별 tsconfig가 extends)
├── eslint.config.ts        # ESLint flat config + typescript-eslint
├── .prettierrc             # JSON 포맷 설정 (+ .prettierignore)
├── vitest.config.ts        # projects 구성 (패키지별 환경 분리 대비)
└── packages/
    └── schema/             # @previs/schema — 블록 스키마 단일 진실 원천
        ├── src/
        │   ├── index.ts            # 공개 API (스키마·타입·파서)
        │   ├── document.ts         # PrevisDocument envelope
        │   ├── block.ts            # BlockBase 공통 규약 + Block 유니온
        │   └── blocks/             # 블록별 zod 스키마 (12파일, 테스트 콜로케이션)
        └── fixtures/               # 샘플 plan/recap JSON (M2 뷰어에서 재사용)
```

설계 결정:

1. **zod 4 discriminated union** — `type` 필드를 판별자로 하는 블록 유니온.
   스키마 선언에서 TypeScript 타입을 자동 추론해 타입 정의·검증 이원화를 방지.
2. **컨테이너 블록 재귀** — tabs/columns는 `blocks: Block[]`를 중첩 포함.
   `z.lazy`로 순환 참조를 처리하고, 문서 레벨 검증에서 최대 중첩 깊이를
   제한해 렌더러의 무한 재귀를 차단.
3. **모든 블록에 `id` 필수** — 코멘트 블록 앵커(ARCHI.md §9)의 전제 조건.
   M1에서 계약으로 고정해야 M6에서 스키마 변경 없이 코멘트를 붙일 수 있다.
4. **`schemaVersion` 필드** — additive-only 진화(AGENTS.md 데이터 규칙)를
   문서 단위로 추적. M1은 `1`로 시작.
5. **스키마는 구조 검증만 담당** — Budget(탭 3~8개, ~150줄)·시크릿 마스킹·
   Grounding 판정은 스킬 계층(M4)의 책임. 단, 구조화 블록의 `inferred` 표기는
   스키마 필드로 지원해 렌더러가 시각 구분할 수 있게 한다.
6. **빌드는 plain tsc** — `dist/`에 ESM + `.d.ts` 산출. 번들러 의존성 없이
   Vite(뷰어)·Node(스킬/런처) 양쪽에서 소비 가능.

### 2. `packages/schema` 패키지 구성

**Files**: `packages/schema/package.json`, `packages/schema/tsconfig.json`

- 패키지명 `@previs/schema`, `type: module`, `exports`는 `dist/` 산출물 지정.
- 의존성은 `zod` 단 하나. devDependencies는 루트에서 공유.
- 스크립트: `build`(tsc), `typecheck`(tsc --noEmit).

### 3. 블록 공통 규약과 유니온

**File**: `packages/schema/src/block.ts`

- `BlockBase`: `id`(비어있지 않은 string — 문서 내 유일성은 문서 레벨 검증),
  `type`(판별자).
- `blockSchema`: 12종 스키마의 discriminated union. 컨테이너 블록의 재귀
  참조 때문에 `z.lazy` 기반으로 구성.
- 알 수 없는 `type`은 파싱 실패 — 전방 호환은 `schemaVersion` 상승으로 다룬다.

### 4. 블록별 스키마 12종

**Files**: `packages/schema/src/blocks/*.ts` (블록당 1파일 + 콜로케이션 테스트)

필드 상세는 아래 Type Definitions 참조. 요점:

- **leaf 블록 (10종)**: prose, callout, file-tree, diff, annotated-code,
  data-model, api-endpoint, wireframe, diagram, question-form
- **컨테이너 블록 (2종)**: tabs, columns — `blocks: Block[]` 중첩
- 구조화 블록(file-tree, data-model, api-endpoint)의 항목에 선택적
  `inferred?: boolean` — Grounding Rule의 "inferred 표기"를 스키마가 지원.
- wireframe의 HTML 조각 sanitize는 렌더러(M3) 책임. 스키마는 string 구조만
  검증하고, `--wf-*`·hex 금지 등 콘텐츠 규칙 lint는 M3/M4에서 별도 검토.

### 5. 문서 envelope

**File**: `packages/schema/src/document.ts`

- `previsDocumentSchema`: `schemaVersion`(literal 1), `id`, `kind`
  (`plan | recap`), `title`, `createdAt`(ISO 8601 string), `blocks`(Block[],
  최소 1개), 선택 `source`(branch / commitRange / pr 등 recap 출처 메타).
- 문서 레벨 refinement: ① 블록 `id` 전역 유일성(중첩 포함) ② 컨테이너 최대
  중첩 깊이 4 제한.

### 6. 공개 API

**File**: `packages/schema/src/index.ts`

- 모든 블록 스키마·추론 타입·`previsDocumentSchema` re-export.
- `parsePrevisDocument`(throw) / `safeParsePrevisDocument`(result 반환) 헬퍼.
- 소비자 시나리오: 스킬(M4)은 발행 전 검증, 뷰어(M2)는 로드 시 safe 파싱.

### 7. 테스트와 픽스처

## Technical Considerations

- **Pattern Usage**: ARCHI.md §5 원칙 준수 — JSON 블록 배열(원칙 1),
  블록 렌더러 매핑을 전제로 한 타입 판별 구조(§8).
- **블록 스키마 영향**: 이 계획 자체가 스키마의 최초 고정. ARCHI.md §8의
  최소 블록 셋 12종과 GOAL.md M1 목록이 일치함을 확인했다.
- **Grounding**: 스키마는 구조만 검증하고 Grounding 판정은 스킬(M4) 책임.
  단 `inferred` 필드로 표기 계약을 미리 고정.
- **시각 계층**: 해당 없음(UI 없음). 단 wireframe 블록 필드(`surface`
  프리셋 enum)는 콘텐츠 규칙의 `--wf-*` 체계 어휘를 따른다.
- **Supabase 영향**: 없음. 단 M6 documents 테이블이 이 envelope를 그대로
  저장할 수 있도록 `schemaVersion`·`kind`를 최상위 필드로 배치.
- **성능**: 검증은 발행/로드 시 1회. 대형 문서도 zod 파싱은 ms 단위 —
  별도 최적화 불필요. 렌더링 성능(가상화 등)은 M2 관심사.
- **Edge Cases**: 빈 blocks 배열(거부), 블록 id 중복(거부), 과도한 중첩
  (깊이 4 초과 거부), 알 수 없는 블록 type(거부), 잘못된 ISO 날짜(거부).
- **버전 리스크**: TypeScript 7.0은 typescript-eslint 지원 범위 밖 —
  6.0.3으로 고정하고 지원 확장 시 후속 사이클에서 상향.

## Type Definitions

문서 envelope:

| 필드 | 타입 | 비고 |
|------|------|------|
| schemaVersion | 1 (literal) | additive 진화 추적 |
| id | string | 문서 식별자 |
| kind | 'plan' \| 'recap' | |
| title | string | |
| createdAt | string (ISO 8601) | |
| source? | { branch?, commitRange?, pr? } | recap 출처 메타 |
| blocks | Block[] (min 1) | |

블록 공통: `{ id: string, type: <판별자> }`. 블록별 고유 필드:

| type | 고유 필드 (요약) |
|------|------------------|
| prose | `markdown: string` — 자유 서술 허용 블록 |
| callout | `variant: info\|warning\|danger\|success`, `title?`, `markdown` |
| file-tree | `entries: [{ path, status?: added\|modified\|deleted\|renamed, from?(renamed 원경로), note?, inferred? }]` |
| tabs | `items: [{ label, blocks: Block[] }]` (min 1) — 재귀 |
| columns | `items: [{ blocks: Block[] }]` (2~4개) — 재귀 |
| diff | `title`, `file`, `lang?`, `diff: string`(unified), `note?` |
| annotated-code | `file?`, `lang`, `code`, `startLine?`, `annotations: [{ line, markdown }]` |
| data-model | `entities: [{ name, fields: [{ name, type, note?, inferred? }], inferred? }]`, `relations?: [{ from, to, kind: 1-1\|1-n\|n-m, label? }]` |
| api-endpoint | `method: GET\|POST\|PUT\|PATCH\|DELETE`, `path`, `summary?`, `request?: { params?: ApiField[], query?: ApiField[], body?: ApiField[] }`, `responses: [{ status: int(100~599), note?, body?: ApiField[] }] (min 1)`, `inferred?` |
| wireframe | `surface: browser\|desktop\|mobile\|popover\|panel`, `title?`, `html: string` |
| diagram | `engine: 'mermaid'`, `code`, `caption?` |
| question-form | `questions: [{ id, prompt, options?: [{ label, description? }], allowFreeText? }]` |

공유 타입:

- `ApiField`: `{ name: string, type: string, required?: boolean, note?: string, inferred?: boolean }` —
  api-endpoint의 params/query/body/응답 body가 공유하는 필드 서술 단위.
- `annotated-code.annotations[].line`: **`code` 발췌 기준 1-base 상대 행**.
  `startLine`은 원본 파일에서 발췌가 시작하는 행 번호(표시용)로, 앵커 계산에
  쓰지 않는다. 검증: `1 <= line <= code의 줄 수` (블록 레벨 refinement).

세부 필드 검증 규칙(빈 문자열 금지, 배열 최소 길이 등)은 구현 시 확정하되
위 계약을 벗어나지 않는다.

## Performance & Cost Impact

- 런타임 영향 없음(라이브러리 패키지). zod 4는 뷰어 번들에 포함될 예정이나
  단일 의존성으로 수용 가능한 크기.

## Backward Compatibility

- 최초 스키마이므로 마이그레이션 없음. 이후 변경은 `schemaVersion` 상승 +
  additive 필드 추가만 허용 (AGENTS.md 데이터 규칙과 정합).

## Test Impact

- 기존 테스트 없음 — 이 사이클이 테스트 인프라(vitest) 자체를 도입한다.
- 신규 테스트 대상: 12종 블록 valid/invalid 파싱, envelope 필수 필드,
  블록 id 전역 유일성, 컨테이너 중첩 깊이 제한, 알 수 없는 type 거부,
