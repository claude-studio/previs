# 와이어프레임 렌더러 — 손그림 계층 1/2 (M3) Implementation Plan

> 버전: 0.3.1 (M3 사이클 1/2 — GOAL.md TRIP 운영 규칙: 사이클 단위 patch.
> M3 완료는 다음 사이클 0.4.0)
> 범위: GOAL.md M3 중 wireframe 렌더러 (사용자 확정: 2사이클 분할 — wireframe
> 먼저. diagram·annotated-code·data-model·api-endpoint·question-form(읽기
> 전용)은 다음 사이클. rough.js 오버레이는 wireframe 블록에만 적용)

## Overview

previs의 정체성인 `--wf-*` 손그림 체계의 1차 구현으로, M2에서 fallback
처리된 wireframe 블록의 실제 렌더러를 구축한다. 콘텐츠 계층 전용 `--wf-*`
테마 토큰(라이트/다크 자동 반전), 시맨틱 HTML sanitize(DOMPurify), surface
프리셋 5종(browser/desktop/mobile/popover/panel) 프레임 크롬, rough.js
스케치 오버레이를 구현한다. 완료 기준: 픽스처 wireframe 블록이 5종 surface
전부에서 손그림 룩으로 렌더링되고 라이트/다크가 자동 반전된다.

## Problem Statement

- wireframe 블록은 스키마(M1)와 픽스처는 존재하지만 뷰어에서 "M3에서
  지원됩니다" placeholder로만 표시된다. plan 문서의 핵심 가치(구현 전 UI
  형태 검토)가 아직 작동하지 않는다.
- AGENTS.md 콘텐츠 규칙 5가 정의한 `--wf-*` 체계는 어휘(토큰 프리픽스·
  surface enum·금지 사항)만 있고 구체 토큰 셋·스타일 명세가 코드 어디에도
  없다. 이번 사이클이 그 단일 정의 지점을 만든다.
- 와이어프레임 HTML 조각은 신뢰 경계 밖 입력(에이전트/파일 열기)이므로
  sanitize 없이는 렌더링할 수 없다 (ARCHI §15).

## Solution Architecture

```
apps/viewer/src/components/blocks/
├── BlockRenderer.tsx            # (modify) wireframe 매핑 추가
├── FallbackBlock.tsx            # (modify) wireframe 라벨 제거
└── wireframe/                   # (new) 콘텐츠 계층 — 손그림 체계
    ├── WireframeBlock.tsx       # surface 프레임 + sanitize HTML + 오버레이 조립
    ├── sanitize.ts              # DOMPurify allowlist 정책 (순수 함수)
    ├── rough-overlay.ts         # 대상 수집·지오메트리·rough.js 그리기 (순수 로직 분리)
    └── wireframe.css            # --wf-* 토큰 + .wf-root 요소 스타일 + surface 크롬
```

설계 결정:

1. **`--wf-*` 토큰 층 — 콘텐츠 계층의 단일 정의 지점.** `wireframe.css`에
   라이트 기본값과 `.dark` 반전값을 정의한다(기존 `html.dark` 클래스 방식
   재사용 — index.css §semantic 토큰과 동일 패턴, 그러나 DESIGN.md 브랜드
   토큰과 값을 공유하지 않는다 — 시각 계층 분리). 토큰 셋(초안):
   `--wf-canvas`(와이어프레임 종이 배경), `--wf-surface`(요소 채움),
   `--wf-ink`(텍스트·스트로크), `--wf-line`(보조선), `--wf-muted`(보조
   텍스트), `--wf-accent`(강조 1색), `--wf-chrome`(surface 프레임 크롬 채움).
   hex 리터럴은 이 토큰 정의 CSS에만 존재한다 — 콘텐츠 HTML의 hex 금지
   규칙(AGENTS.md)과 충돌하지 않는다(룩은 렌더러 소유).
2. **시맨틱 요소 스코프 스타일.** `.wf-root` 스코프에서 시맨틱 태그
   (h1~h4, p, header, nav, main, section, aside, footer, form, label,
   input, textarea, select, button, table, ul/ol/li, a, figure, hr 등)에
   요소 셀렉터로 와이어프레임 스타일을 입힌다. 에이전트 HTML은 클래스 없이
   시맨틱 구조만으로 손그림 룩을 얻는다 — "룩은 렌더러가 소유한다"의 구현.
   손그림 미학의 타이포는 콘텐츠 계층 전용 폰트 스택(기존 자체 호스팅 폰트
   범위 내 캐주얼 조합)으로 앱 크롬과 구분한다. 새 폰트 패키지 추가는
   하지 않는다(추가가 필요하면 별도 결정).
3. **Sanitize 정책 — 태그·속성 이중 allowlist.** DOMPurify로 시맨틱 구조·
   텍스트·폼 태그만 허용하고, 속성도 명시적 allowlist로만 통과시킨다
   (DOMPurify 기본 속성 allowlist는 width/height/src/action 등 표현·지오
   메트리·리소스 속성을 남기므로 그대로 쓰지 않는다):
   - `ALLOWED_TAGS` allowlist (위 2의 시맨틱 셋). script/style/iframe/svg/
     math 등은 진입 불가.
   - `ALLOWED_ATTR` 명시 allowlist — 구조·접근성 최소 셋(예: `alt`,
     `colspan`, `rowspan`, `placeholder`, `type`, `for`, `checked`,
     `selected`, `disabled`) 만 허용. `style`·`class`·`width`·`height`·
     `src`·`href`·`action`·`formaction`·`background`·`coords` 등은
     allowlist 밖이므로 자동 제거 — 좌표·색·폰트 하드코딩 금지(콘텐츠
     규칙)와 외부 요청 차단을 기계적으로 강제한다. `ALLOW_DATA_ATTR:
false`로 data-* 도 차단하고, 레이아웃 힌트가 필요해지면 후속
     사이클에서 `data-wf-*`만 additive로 연다.
   - 이벤트 핸들러(on*)·`javascript:` URL은 DOMPurify 기본 제거.
   - **폼·버튼 view-only 강제**: sanitize 후처리에서 폼 컨트롤(button,
     input, textarea, select)에 `disabled`를 부여하고, WireframeBlock이
     `.wf-root` 래퍼에서 submit·click 기본 동작을 위임 차단한다 — 기본
     폼 제출이 SPA를 리로드해 메모리 문서(열린 파일)를 날리는 경로를
     봉쇄. disabled 룩은 wireframe.css가 일반 컨트롤과 동일하게 통일.
   - `<a>`는 href가 allowlist 밖이라 제거된다 — 와이어프레임 안 링크는
     시각 요소일 뿐 실제 이동 경로가 아니다.
   - `<img>`는 src 제거 후 alt 텍스트를 담은 placeholder 박스로 렌더 —
     외부 리소스 요청 차단(데이터 주권, ARCHI §15).
   - sanitize 결과가 빈 문자열이면 "표시할 수 없는 와이어프레임" 인라인
     안내를 렌더(문서 전체 렌더링은 깨지지 않게).
4. **Surface 프리셋 5종 — 렌더러 소유 크롬.** 프레임 크롬(상단 바·창
   컨트롤·노치 등)과 논리 폭 프리셋은 렌더러가 소유한다(콘텐츠의 폭/좌표
   하드코딩 금지와 구분되는 정당한 렌더러 영역):
   - `browser`: 창 컨트롤 점 3개 + 주소창 스트립, 문서 본문 폭.
   - `desktop`: 타이틀 바 창 프레임, 문서 본문 폭.
   - `mobile`: 좁은 논리 폭(중앙 정렬) + 큰 라운드 + 상단 스피커 힌트.
   - `popover`: 좁은 카드 + 그림자 없는 스케치 테두리 강조.
   - `panel`: 세로형 사이드 패널 프레임(한쪽 정렬).
     `title`(optional)은 크롬 영역에 표시한다.
5. **rough.js 스케치 오버레이 — wireframe 한정.** 컨테이너에 absolute SVG
   오버레이를 얹고, 프레임과 박스형 요소(section, header, button, input,
   table 등)의 boundingRect를 측정해 rough.js 사각형/선을 그린다. 실제 CSS
   테두리는 희미하게(또는 투명) 두고 rough 스트로크가 시각 테두리를 담당:
   - **결정성**: rough.js `seed`를 block.id 결정적 해시로 유도(card-identity
     패턴 재사용) — 리렌더마다 스케치가 흔들리지 않는다.
   - **테마 반전**: 스트로크·채움 색은 그리기 후 `var(--wf-ink)` 등 CSS 변수
     참조로 지정해 테마 전환 시 재그리기 없이 자동 반전한다.
   - **재그리기 트리거**: ResizeObserver(컨테이너 폭 변화)와 block 내용
     변경 시에만.
   - **로직 분리**: 대상 요소 수집·지오메트리 계산·시드 유도는 DOM 측정
     입력을 받는 순수 함수로 분리해 jsdom에서 단위 테스트 가능하게 한다
     (jsdom은 boundingRect가 0이므로 컴포넌트 테스트는 구조·sanitize 중심,
     시각 확인은 수동).
6. **픽스처 보강 — 스키마 변경 없음.** kitchen-sink의 wireframe 블록을
   시맨틱 태그가 풍부한 예제로 확장하고 surface 5종을 모두 시연하도록 블록을
   추가한다(kitchen-sink는 전 기능 시연 목적). sample-plan.json에도 plan
   문서의 실제 사용례에 해당하는 wireframe 블록 1개를 추가한다.

## Implementation Details

### 1. `--wf-*` 토큰과 요소 스타일 (콘텐츠 계층 CSS)

**File**: `apps/viewer/src/components/blocks/wireframe/wireframe.css` (new)

- `:root`에 `--wf-*` 라이트 값, `.dark`에 반전 값 정의 (설계 결정 1의 토큰
  셋). index.css의 semantic 토큰과 같은 메커니즘, 다른 값 공간.
- `.wf-root` 스코프 요소 스타일: 시맨틱 태그별 와이어프레임 표현
  (버튼=스케치 박스+채움, input=밑줄 박스, 이미지 placeholder=대각선 박스,
  표=스케치 그리드 등). 모든 색은 `--wf-*` 변수만 참조.
- surface 크롬 스타일 5종 (설계 결정 4).
- `WireframeBlock.tsx`에서 import (vite CSS 번들링 — index.css 수정 불필요.
  단, Tailwind `@theme` 밖의 plain CSS이므로 유틸 클래스 충돌 없음).

### 2. sanitize 정책

**File**: `apps/viewer/src/components/blocks/wireframe/sanitize.ts` (new)

- `sanitizeWireframeHtml(html: string): string` — DOMPurify 설정(설계 결정
  3)을 캡슐화한 순수 함수. `ALLOWED_TAGS`·`ALLOWED_ATTR` 상수를 export해
  테스트가 정책을 직접 검증할 수 있게 한다. `ALLOW_DATA_ATTR: false`.
- `<img>` placeholder 치환·폼 컨트롤 `disabled` 부여는 DOMPurify hook 또는
  후처리로 같은 모듈에서 수행 — 정책이 한 파일에 응집.

### 3. rough.js 오버레이

**File**: `apps/viewer/src/components/blocks/wireframe/rough-overlay.ts` (new)

- 순수 로직: 스케치 대상 셀렉터 정의, (컨테이너 기준) rect 목록 →
  rough.js 도형 명세 변환, block.id → seed 해시.
- DOM 어댑터: SVG 생성·`rough.svg()` 그리기·CSS 변수 색 참조 후처리·
  ResizeObserver 연결. WireframeBlock에서 effect로 호출.
- `ResizeObserver`는 feature-detect — 없는 환경(jsdom)에서는 라이브
  재그리기만 생략하고 크래시 없이 동작한다. 테스트 환경에는
  `src/test-setup.ts`에 결정적 stub을 추가한다.

### 4. WireframeBlock 컴포넌트

**File**: `apps/viewer/src/components/blocks/wireframe/WireframeBlock.tsx` (new)

- surface 프리셋 크롬(+ title) → `.wf-root`에 sanitize된 html 주입
  (`dangerouslySetInnerHTML` — sanitize 통과분만, DiffBlock의 shiki 주입과
  동일한 경계 패턴) → rough 오버레이 effect.
- `.wf-root` 래퍼에서 submit·click 기본 동작을 위임 차단 (설계 결정 3의
  view-only 강제 — 내비게이션/리로드 회귀 테스트 대상).
- sanitize 결과 빈 문자열이면 인라인 안내 렌더 (설계 결정 3).
- **default export**로 노출해 `React.lazy` 진입점이 되게 한다.

## Technical Considerations

- **Pattern Usage**: 블록 렌더러 매핑(ARCHI §8) 재사용, lazy 로직 분리는
  DiffBlock의 shiki 패턴과 동형. 결정적 해시는 card-identity 패턴 재사용.
- **블록 스키마 영향**: 없음 — wireframe 스키마(M1 확정)를 소비만 한다.
  픽스처 변경은 기존 스키마 검증 테스트가 보호한다.
- **Grounding**: 해당 없음 (렌더링만). 단, sanitize의 style/class 제거는
  콘텐츠 규칙(hex·좌표 금지)을 렌더러 쪽에서 기계적으로 강제하는 장치.
- **시각 계층**: 이 계획의 본질 — `--wf-*`는 블록 콘텐츠 계층이다. DESIGN.md
  토큰(index.css semantic 층)과 값·변수를 공유하지 않는다. wireframe/ 폴더
  밖으로 `--wf-*`가 새지 않게 하고, rough.js는 사용자 확정대로 wireframe
  블록에만 적용한다.
- **Supabase 영향**: 없음 (로컬 전용).
- **성능**: rough.js(~9KB gzip 수준)와 dompurify는 wireframe 블록 렌더
  경로에서만 필요 — WireframeBlock을 dynamic import(React.lazy)로 분리해
  wireframe 없는 문서의 초기 로드에 영향을 주지 않는다. 오버레이 재그리기는
  ResizeObserver 단위로 제한(스크롤·테마 전환에는 그리지 않음).
- **Edge Cases**: sanitize 후 빈 결과(인라인 안내), 알 수 없는 태그 대량
  포함 HTML(제거 후 잔여만 렌더), 중첩 인터랙티브 요소(무력화된 시각
  요소로만 렌더), 매우 긴 와이어프레임(오버레이는 컨테이너 단위 1개 SVG —
  요소 수에 비례한 도형 수는 rough.js가 감당 가능한 수준, 픽스처 규모 기준),
  jsdom 환경(rect 0·`ResizeObserver` 부재 — feature-detect로 오버레이
  미도출이 크래시로 이어지지 않게 방어), 폼 제출로 인한 SPA 리로드(위임
  차단으로 봉쇄 — 메모리 문서 보호).
- **접근성**: sanitize가 시맨틱 구조를 보존하므로 스크린 리더가 와이어프레임
  구조를 읽을 수 있다. 오버레이 SVG는 `aria-hidden` 장식 요소로 처리.

## Performance & Cost Impact

- 번들: roughjs+dompurify는 WireframeBlock lazy 청크로 격리 — 초기 로드
  불변. 렌더: 오버레이 그리기는 블록당 1회 + 리사이즈 시 재그리기.

## Backward Compatibility

- 스키마·기존 렌더러 계약 변경 없음 (additive). 기존 문서의 wireframe
  블록은 placeholder → 실제 렌더로만 바뀐다. 픽스처 변경도 스키마 유효
  범위 내 콘텐츠 수정이다.
