# 잔여 블록 렌더러 — 손그림 계층 2/2 (M3 완료) Implementation Plan

> 버전: 0.4.0 (M3 사이클 2/2 — GOAL.md TRIP 운영 규칙: 마일스톤 완료 시 minor
> 상승. 이전 사이클 0.3.1 계획 문서가 "M3 완료는 다음 사이클 0.4.0"으로 예고)
> 범위: diagram(mermaid)·annotated-code·data-model·api-endpoint·question-form
> 렌더러 5종 (사용자 확정: question-form은 읽기 전용으로 포함, diagram은
> mermaid handDrawn 룩 + `--wf-*` 색 매핑, data-model은 커스텀 엔티티 카드)

## Overview

M2에서 fallback("M3에서 지원됩니다")으로 남은 블록 5종의 실제 렌더러를
구현해 M3(시각화 심화 — 손그림 계층)를 완료한다. diagram은 mermaid
handDrawn 룩으로 손그림 정체성을 잇고, annotated-code는 shiki 하이라이트에
마진 주석을 얹으며, data-model·api-endpoint·question-form은 구조화 사실
블록으로 카드형 렌더링한다. 완료 기준(GOAL.md M3): 전체 블록 셋 렌더링,
라이트/다크 자동 반전 — kitchen-sink 픽스처가 fallback 없이 전부 렌더된다.

## Problem Statement

- 블록 5종은 스키마(M1)·픽스처가 존재하지만 뷰어에서 placeholder로만
  표시된다. plan 문서의 다이어그램·미해결 질문, recap 표준 골격의
  스키마/API 변경 섹션(AGENTS.md 콘텐츠 규칙 2)이 아직 시각화되지 않는다.
- diff 하이라이터(shiki)가 DiffBlock 모듈에 갇혀 있어 annotated-code가
  재사용할 공용 진입점이 없다.
- BlockRenderer의 lazy 로딩 경계(에러 바운더리 + Suspense)가 wireframe
  전용으로 하드코딩돼 있어 두 번째 lazy 블록(diagram)이 중복을 만든다.

    ├── FallbackBlock.tsx            # (modify) 미지 타입 방어용으로 문구 정리
    ├── DiffBlock.tsx                # (modify) 공용 하이라이터로 위임

설계 결정:

1. **시각 계층 배속 — diagram만 손그림 콘텐츠 계층.** M3 손그림 계층의
   본질은 "화면·구조 시각화" 재료다. diagram은 wireframe과 같은 콘텐츠
   계층으로서 `--wf-*` 토큰 색·폰트를 사용한다(사용자 확정). 반면
   annotated-code·data-model·api-endpoint·question-form은 diff·file-tree와
   같은 "구조화 사실" 계열이므로 기존 블록 크롬 관례(DESIGN.md 앱 토큰,
   DiffBlock 동형의 카드 섹션)를 따른다. 근거: M2에서 diff가 앱 토큰으로
   확정 출하된 선례 — 사실 기술 블록까지 손그림화하면 가독성이 미학에
   밀린다. `--wf-*` 변수는 계속 wireframe/디렉토리와 diagram 렌더러만
   참조한다.
2. **DiagramBlock — mermaid handDrawn, 렌더러 소유 룩.**
   - mermaid 11의 `look: 'handDrawn'`을 켜고 `theme: 'base'` +
     `themeVariables`로 색을 지정한다. mermaid 테마 엔진은 파생색 계산
     때문에 CSS 변수 문자열을 받지 못하므로, 렌더 시점에
     `getComputedStyle`로 `--wf-*` 실값을 읽어 전달한다(와이어프레임의
     CSS 변수 참조 방식과 다른 이유를 코드가 아닌 이 계획에 기록).
   - **`--wf-*` 토큰 분리 로딩**: 토큰 정의는 현재 wireframe.css에 있고
     WireframeBlock의 lazy import로만 로드된다 — diagram만 있는 문서에서
     빈 값을 읽는 문제를 막기 위해, 토큰 정의(`:root`·`.dark` 블록)를
     `wireframe/tokens.css`로 추출하고 wireframe.css와 DiagramBlock이
     각각 import한다 (번들러가 중복 제거).
   - **공용 mermaid 어댑터 — 전역 설정·동시성 계약.** mermaid 설정은
     전역(site-wide)이고 뷰어는 StrictMode라 이펙트가 이중 실행되므로,
     블록별 설정(seed·테마 실값)을 안전하게 쓰기 위해 렌더 호출을
     직렬화하는 어댑터 모듈을 둔다: 단일 initialize(securityLevel 등
     보안 설정·startOnLoad 비활성만 전역), 렌더 큐, 시도별 고유 DOM-safe
     렌더 id(block.id 직접 사용 금지 — StrictMode 중복 시도·특수문자
     충돌 회피), 테마 세대 가드(관찰된 테마가 렌더 완료 전에 바뀌면
     stale SVG 커밋을 폐기).
   - **호출별 설정 주입 — 신뢰 frontmatter.** mermaid `render` API에는
     설정 인자가 없으므로, 호출별 설정(theme base·themeVariables 실값·
     look handDrawn·handDrawnSeed)은 작성자 지시어를 strip한 본문 앞에
     렌더러가 생성한 YAML frontmatter를 주입해 적용한다. securityLevel
     등 보안 설정은 frontmatter로 변경할 수 없으므로(mermaid secure
     목록) 전역 initialize에만 둔다 — 신뢰 설정 주입과 보안 설정 고정이
     분리된다.
   - **테마 반전**: ThemeToggle이 `html.dark` 클래스를 토글하므로
     DiagramBlock이 documentElement class를 MutationObserver로 관찰해
     토큰 실값을 다시 읽고 재렌더한다. 앱 상태 리프팅(컨텍스트) 없이
     렌더러 로컬로 해결 — 시각 계층 분리 유지.
   - **결정성**: mermaid `handDrawnSeed`를 block.id 결정적 해시로 유도
     (card-identity 패턴 재사용, wireframe rough 시드와 동일 원칙).
     DOM 렌더 id는 시도별 고유값, 스케치 seed는 블록별 결정값 — 역할이
     다르다.
   - **보안 — strict 유지 + 사전 입력 거부 + 출력 SVG 후처리.** diagram
     코드는 신뢰 경계 밖 입력이다. mermaid `securityLevel`은 기본값
     `'strict'`를 유지하고 `'loose'`로 낮추지 않는다. 단, strict는 HTML
     라벨·클릭 동작 억제가 중심이고 이미지 노드 등 외부 URL은 렌더
     과정(레이아웃 측정)에서 이미 로드될 수 있으므로 출력 후처리만으로는
     네트워크 요청을 막지 못한다. 따라서 **렌더 전** strip된 본문에
     보수적 거부 정책을 적용한다 — 다음 중 하나라도 매치되면
     `mermaid.render`를 호출하지 않고 인라인 안내("외부 리소스를
     참조하는 다이어그램은 지원되지 않습니다") + 원본 코드로 렌더한다:
     (a) `@{…}` 셰이프 메타데이터 전체 — 완전한 YAML로 해석되어 인용 키·
     `\u` 이스케이프·anchor/alias 로 자산 키(img/icon)를 난독화할 수
     있으므로 raw 문자열 검사 대신 블록 자체를 거부한다(Codex 리뷰에서
     우회 3종 확인 후 강화), (b) CSS `url(` 함수 표기(style·classDef
     포함), (c) URL 스킴 리터럴(`://`, `data:` 등)과 protocol-relative
     `//`, (d) 아이콘 팩 축약(`fa:` 등). 라벨 텍스트 속 무해한 URL이나
     합법적 shape 지정까지 거부되는 오탐은 감수한다 — 안내에 원본 코드가
     함께 표시되므로 정보 손실이 없고, 완화가 필요해지면 후속 사이클에서
     해석된 YAML 키 allowlist로 additive하게 푼다. 렌더 결과 SVG의
     DOMPurify(SVG 프로파일) 후처리는 심층 방어로 유지해 외부 리소스
     참조·스크립트성 잔여물을 제거한다 — 와이어프레임 img placeholder
     처리와 같은 데이터 주권 원칙 (ARCHI §15). 이미지 노드(인용 키·
     이스케이프·anchor/alias 변형 포함)·`classDef url(...)` 입력이 렌더
     호출·네트워크 요청 없이 안내로 끝나는 회귀 테스트를 둔다.
   - **작성자 설정 오버라이드 차단.** mermaid 코드의 YAML frontmatter와
     `%%{init: …}%%` 지시어는 다이어그램 단위로 테마 등 비보안 설정을
     덮어쓸 수 있다 — "룩은 렌더러가 소유한다" 원칙에 따라 어댑터가
     렌더 전에 frontmatter·init 지시어를 제거(strip)하고 렌더러 생성
     설정만 적용한다.
   - **실패 격리**: mermaid 파싱/렌더 실패 시 문서 전체를 깨지 않고
     블록 자리에 인라인 오류 안내 + 원본 코드 텍스트를 표시한다.
   - **lazy 분리**: mermaid는 매우 큰 의존성이므로 WireframeBlock과 동일한
     React.lazy 경로로 격리한다. `caption`은 figure/figcaption 시맨틱으로.
3. **lazy 경계 공용화.** BlockRenderer의 WireframeErrorBoundary·로딩
   스켈레톤·Suspense 조합을 라벨을 매개변수로 받는 공용 경계로 일반화해
   wireframe·diagram 두 lazy 렌더러가 공유한다 (BlockRenderer.tsx 내부
   리팩토링 — 파일 추가 없음).
4. **shiki 공용 하이라이터 — `lib/highlighter.ts`.** DiffBlock의 모듈
   레벨 싱글턴을 lib로 승격한다. 요청 lang을 동적으로 `loadLanguage`하고,
   shiki가 모르는 lang은 plaintext로 폴백해 렌더 실패가 없게 한다.
   DiffBlock은 공용 모듈에 위임하도록 수정하되 동작(diff lang, github
   라이트/다크 듀얼 테마, 라인 분류 transformer)은 불변. annotated-code도
   동일 듀얼 테마를 사용해 코드 계열 블록의 룩을 통일한다.
5. **AnnotatedCodeBlock — 마진 주석 (GOAL.md M3 명세).**
   - 코드는 공용 하이라이터로 렌더하고, shiki transformer로 주석 대상
     라인에 마커 클래스·앵커를 부여한다. 라인 번호는 `startLine`(기본 1)
     부터 표시하되 annotation의 `line`은 스키마 계약대로 코드 기준
     1-시작 상대 번호로 해석한다.
   - 주석 배치: 넓은 뷰포트에서는 코드 우측 마진 컬럼에 번호 마커로
     대응시키고, 좁은 뷰포트에서는 코드 아래 목록으로 접힌다(반응형
     그리드 — 정밀 픽셀 정렬보다 마커 번호 대응을 계약으로 삼아 레이아웃
     취약성을 줄인다). 주석 본문 markdown은 ProseBlock과 같은
     react-markdown 경로로 렌더한다.
   - `file`(optional)은 DiffBlock 헤더와 동형의 크롬에 표시.
6. **DataModelBlock — 커스텀 엔티티 카드 (사용자 확정).** 엔티티별 카드에
   필드 테이블(name·type·note)을 렌더하고, 필드·엔티티의 `inferred`는
   Grounding Rule의 시각 장치로서 명시적 배지("추론됨")로 표시한다.
   `relations`는 카드 아래 관계 목록으로 from → to, kind 배지(1-1/1-n/n-m),
   label을 표시한다. mermaid erDiagram 변환은 채택하지 않는다 — note·
   inferred가 손실되고 임의 문자열의 mermaid 문법 이스케이프 리스크가
   있다 (사용자 확정 근거).
7. **ApiEndpointBlock — 요청/응답 카드.** 메서드 배지(GET/POST/PUT/PATCH/
   DELETE) + path 코드 표기 + summary 헤더, request의 params/query/body
   섹션별 필드 테이블, responses의 status·note·body 목록. 필드 테이블은
   data-model과 스키마가 거의 같으므로(name·type·note·inferred, api만
   required 추가) `FieldTable` 공유 서브컴포넌트로 통일하고 required
   컬럼은 옵션으로 둔다.
8. **QuestionFormBlock — 읽기 전용 (사용자 확정).** 질문 prompt와 선택지
   label·description을 비인터랙티브 목록으로 렌더한다. 실제 폼 컨트롤을
   쓰지 않아(라디오/입력 없음) 제출·포커스 경로 자체가 없다 — wireframe
   view-only와 같은 원칙. `allowFreeText`는 "자유 입력 허용" 배지로
   표시하고, 응답 수집이 협업 모드(M6) 기능임을 블록 푸터에 안내한다.
9. **FallbackBlock — 미지 타입 방어로 축소.** 5종 라벨 제거로 라벨 맵이
   비므로, 스키마 밖 타입(향후 schemaVersion 상승 대비)의 방어 렌더로
   역할을 정리하고 문구를 "지원되지 않는 블록 타입"으로 바꾼다
   ("M3에서 지원됩니다"는 이번 사이클로 거짓이 된다).
10. **픽스처 보강 — 스키마 변경 없음.** kitchen-sink는 이미 5종 예제를
    담고 있어 렌더러 검증 입력으로 그대로 쓴다. 문서 골격 정합을 위해
    sample-plan에 diagram·question-form(계획 문서의 실사용례 — 흐름도와
    미해결 질문)을, sample-recap에 data-model·api-endpoint·annotated-code
    (recap 표준 골격의 스키마/API 변경 섹션)를 추가한다.

## Implementation Details

### 1. shiki 공용 하이라이터

**File**: `apps/viewer/src/lib/highlighter.ts` (new),
`apps/viewer/src/components/blocks/DiffBlock.tsx` (modify)

- 싱글턴 하이라이터 생성·요청 lang 동적 로딩·미지 lang plaintext 폴백을
  캡슐화. github-light/github-dark 듀얼 테마 상수도 이 모듈이 소유.
- DiffBlock은 하이라이터 취득만 공용 모듈로 위임 — classifyDiffLines·
  라인 분류 transformer·렌더 구조는 그대로 둔다 (기존 테스트 불변).

### 2. AnnotatedCodeBlock

**File**: `apps/viewer/src/components/blocks/AnnotatedCodeBlock.tsx` (new)

- 공용 하이라이터로 코드 렌더 + 주석 라인 마커 transformer.
- annotation ↔ 라인 대응·표시용 라인 번호 계산(startLine 반영)은 DOM
  없이 검증 가능한 순수 함수로 분리해 콜로케이션 테스트한다 (DiffBlock의
  classifyDiffLines 선례).
- 마진 주석 반응형 배치(설계 결정 5), 주석 markdown은 react-markdown.
- 하이라이트 완료 전에는 DiffBlock과 동형의 plain `<pre>` 폴백을 표시.

### 3. DiagramBlock (lazy) + mermaid 어댑터 + 토큰 분리

**Files**: `apps/viewer/src/components/blocks/DiagramBlock.tsx` (new),
`apps/viewer/src/components/blocks/diagram-mermaid.ts` (new),
`apps/viewer/src/components/blocks/wireframe/tokens.css` (new),

## Technical Considerations

- **Pattern Usage**: 블록 렌더러 매핑(ARCHI §8), React.lazy 격리·결정적
  시드·순수 로직 분리(wireframe 선례), shiki 듀얼 테마(DiffBlock 선례)를
  재사용한다. lazy 경계 공용화는 두 번째 사용처가 생기는 시점의 정당한
  일반화다.
- **블록 스키마 영향**: 없음 — 5종 모두 M1 확정 스키마를 소비만 한다.
  픽스처 변경은 기존 스키마 검증 테스트가 보호한다.
- **Grounding**: 렌더러는 표시만 하지만, data-model·api-endpoint의
  `inferred`를 명시 배지로 노출해 Grounding Rule의 "inferred 표기"가
  뷰어에서 시각적으로 성립하게 한다.
- **시각 계층**: diagram = 콘텐츠 계층(`--wf-*`), 나머지 4종 = 앱 계층
  (DESIGN.md 토큰) — 설계 결정 1에 근거 명시. rough.js 오버레이는 이전
  사이클 사용자 확정대로 wireframe 전용을 유지한다(diagram의 손그림은
  mermaid handDrawn이 담당).
- **Supabase 영향**: 없음 (로컬 전용). question-form 응답 수집은 M6 범위.
- **성능**: mermaid는 lazy 청크로 격리해 초기 로드 불변. shiki lang은
  요청 시 동적 로딩·하이라이터 싱글턴 재사용. 다이어그램 재렌더는 테마
  전환·블록 내용 변경 시로 제한. 구조화 카드 3종은 정적 렌더로 비용 없음.
- **Edge Cases**: mermaid 문법 오류(인라인 안내 + 원본 코드), 미지 shiki
  lang(plaintext 폴백), 문서 내 다중 diagram·StrictMode 이중 이펙트(어댑터
  직렬화 + 시도별 고유 렌더 id로 충돌 방지), 매우 큰 다이어그램(최대 높이
  + 스크롤), 이미지 노드 등 외부 리소스 참조(렌더 전 거부 + SVG 후처리 심층 방어),
  작성자 frontmatter/init 지시어(strip 후 신뢰 frontmatter 주입), diagram 단독 문서(tokens.css 직접
  import로 토큰 보장), 주석 없는 annotated-code(annotations 빈 배열 —
  코드만 렌더), 테마 고속 연속 전환(세대 가드로 stale SVG 폐기·마지막
  상태로 수렴), jsdom(mermaid 렌더 불가 — 컴포넌트 테스트는 mermaid 모킹,
  순수 함수 중심 검증).
- **접근성**: diagram SVG에 caption 기반 접근 가능한 이름 제공, 구조화
  카드는 시맨틱 테이블/목록 마크업, question-form은 실제 폼 컨트롤이
  없으므로 포커스 트랩·오동작 경로 없음.

## Type Definitions

- 신규 공유 타입 없음 — 블록 타입은 전부 `@previs/schema`에서 import.
- `highlighter.ts`: 하이라이트 요청 옵션(lang·테마) 수준의 모듈 로컬 타입.
- `DiagramBlock`: 토큰 판독 결과·themeVariables 매핑의 모듈 로컬 타입.
- `FieldTable`: 필드 행·컬럼 옵션(required 표시 여부)의 컴포넌트 prop 타입.

## Performance & Cost Impact

- 번들: mermaid는 DiagramBlock lazy 청크로 격리 — diagram 없는 문서의
  초기 로드 불변. shiki lang 추가 로딩은 annotated-code 등장 시에만.
- 렌더: 다이어그램은 블록당 1회 + 테마 전환 시 재렌더. 구조화 카드
  3종은 순수 정적 렌더.

## Backward Compatibility

- 스키마·기존 렌더러 계약 변경 없음 (additive). 기존 문서의 5종 블록은
  placeholder → 실제 렌더로만 바뀐다. DiffBlock 리팩토링은 동작 불변.
- FallbackBlock 문구 변경은 스키마 검증을 통과한 문서에서는 도달 불가
  경로의 방어 문구다.
