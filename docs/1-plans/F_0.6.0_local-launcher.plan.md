# 로컬 런처 — 싱글턴 뷰어 서버 (M5) Implementation Plan

## Overview

세션마다 뷰어 dev 서버가 중복 기동되는 것을 막는 "재사용 우선" 싱글턴 런처를
구현한다. 스킬(`/previs-plan`·`/previs-recap`)이 발행 직후 런처를 호출하면,
런처는 이미 떠 있는 뷰어가 있으면 그 URL을 반환하고 없으면 하나만 기동한다.
다중 세션이 동시에 호출해도 프로세스는 정확히 하나만 살아남는다. M5 완료 시
previs는 "개인 도구"로 실사용 가능한 1차 완성 상태가 된다.

## Problem Statement

현재 뷰어 서버 기동은 전적으로 수동이다.

- `/previs-plan`·`/previs-recap` 스킬은 사용자에게 `pnpm --filter @previs/viewer dev`를
  직접 실행하라고 안내만 한다. 세션마다 사용자가 서버가 떠 있는지 판단해야 하고,
  중복 기동·좀비 프로세스를 막을 장치가 없다.
- 포트 유도 로직(`47738 + hashPath(projectRoot) % 64`)이 `vite.config.ts`에만
  임베드돼 있고, `strictPort: false`라 점유 시 Vite가 전용 대역(`47738~47801`)을
  벗어나 임의 상향할 수 있다 (AGENTS.md의 "대역 내 +1 상향 탐색" 규칙 위반 소지).
- 헬스/정체 확인 엔드포인트(`/api/viewer-info`)가 없어, 특정 포트를 점유한
  프로세스가 "우리 뷰어"인지 판별할 방법이 없다.
- 유휴 자동 종료가 없어 한번 뜬 서버가 무한정 살아 있는다.

AGENTS.md "로컬 뷰어 실행 규칙"과 ARCHI.md §11(설계 확정)이 요구하는 런처
동작이 아직 코드로 존재하지 않는다.

## Solution Architecture

기존 CLI 패키지 패턴(`@previs/schema`·`@previs/recap`: plain `tsc` → `dist/` →
`node ...js`)을 따라 새 패키지 **`@previs/launcher`**를 만든다. 런처는 Vite dev
서버를 백그라운드 detached 프로세스로 기동하는 얇은 오케스트레이터이고, 뷰어
런타임(정체 응답·유휴 종료·락 정리)은 Vite 플러그인으로 dev 서버 프로세스
안에서 동작한다.

핵심 결정(사용자 확정):

1. **서빙 방식 = Vite dev 서버 유지.** `/api/documents`는 기존 플러그인 그대로
   두고, `/api/viewer-info`와 런타임 수명주기만 새 플러그인으로 추가한다.
   dist 정적 서버로 전환하지 않는다.
2. **타깃 범위 = previs 리포 도그푸딩 + `PREVIS_DOCS_DIR` 오버라이드.** 런처는
   기본적으로 previs 리포의 `.previs/`를 서빙하되, 환경변수로 다른 프로젝트의
   `.previs/`를 가리킬 수 있다. 완전 멀티 프로젝트(프로젝트별 별도 인스턴스)는
   M7 패키징 이후 과제로 둔다.
3. **스킬 연동 포함.** 두 스킬의 수동 `pnpm dev` 안내를 런처 호출로 교체한다.

싱글턴 보장은 **원자적 락 파일 + `/api/viewer-info` 헬스체크**의 조합이다.

- **락**: `fs.open(lockPath, 'wx')`로 원자적 배타 생성. 락 파일은 서버 수명 동안
  유지되며 `{ pid, port, startedAt }`를 기록한다(AGENTS.md "확정 포트는 락 파일에
  기록"). 서버 종료 시 스스로 삭제한다.
- **재사용 판정**: 락이 있으면 기록된 포트로 `/api/viewer-info`를 조회해
  정체(name)와 타깃(docsDir)이 일치하면 그 URL을 재사용한다.
- **stale 복구**: 락은 있으나 헬스 응답이 없고 기록된 pid가 죽어 있으면
  (`process.kill(pid, 0)`) 락을 stale로 보고 제거 후 재취득한다.
- **동시 기동 경쟁**: `wx`가 EEXIST면 다른 런처가 기동 중이므로, 헬스가 뜰
  때까지 바운드된 폴링 후 재사용한다.

포트는 전용 대역 안에서 런처가 권위를 갖는다. 후보 포트(`47738 + hash % 64`)가
외부 프로세스에 점유돼 있으면 대역 `[47738, 47801]` 안에서 +1 상향 탐색하고,
확정 포트를 `PREVIS_PORT`로 넘겨 Vite를 `strictPort: true`로 기동한다(대역
이탈 방지). 대역이 전부 점유되면 명확한 에러로 종료한다.

## Implementation Details

### 1. 신규 패키지 `@previs/launcher` — 경로/포트 유도

**File**: `packages/launcher/src/paths.ts` (new)

포트·락·로그·docsDir 유도를 단일 진실 원천으로 모은다. 현재
`vite.config.ts`에 있는 `hashPath`/`viewerPort` 로직을 이곳으로 이전하고,
`vite.config.ts`는 이 모듈을 상대 경로로 import해 폴백 포트 계산에 재사용한다
(중복 제거).

- `hashPath(value)` — djb2 해시 (현행 로직 그대로).
- `viewerPort(targetKey)` — `47738 + (hashPath(targetKey) % 64)`. 대역 상수
  `PORT_START=47738`, `PORT_COUNT=64`, `PORT_END=47801`를 export.
- `resolveTarget(env, projectRoot)` — docsDir와 `targetKey` 결정:
  `PREVIS_DOCS_DIR`가 있으면 그 값을, 없으면 `<projectRoot>/.previs`를 canonical
  docsDir로 삼는다. **`projectRoot`는 호출자가 명시 전달**한다(cwd 의존 금지):
  런처는 리포 루트를, Vite는 이미 계산해 둔 `../..`(리포 루트)를 넘긴다.
  filtered dev(`pnpm --filter @previs/viewer dev`)의 cwd가 `apps/viewer`라
  cwd로 유도하면 어긋나기 때문이다(P2-D 후속).
  - **미존재 디렉토리 안전 정규화** — 최초 발행 전에는 `.previs/`가 없을 수
    있으므로 `fs.realpath`를 leaf에 바로 걸지 않는다. `path.resolve`로 절대
    정규화하고, 심링크 해소가 필요하면 **존재하는 가장 가까운 상위**만
    realpath한 뒤 나머지를 이어붙인다(throw 금지).
  **`targetKey` = 이 canonical docsDir 문자열** — 포트·락 파생, 헬스 응답의
  `docsDir`, 재사용 비교가 모두 동일한 단일 키를 쓴다(정체 일관성). "무엇을
  서빙하는가"가 곧 싱글턴 정체가 된다.
- `lockPath(targetKey)` — `os.tmpdir()`에 `previs-viewer-<hash>.lock`. 리포
  안에 두지 않아 git 오염을 피한다.
- `logPath(targetKey)` — detached 서버의 stdout/stderr 리다이렉트 대상
  (`os.tmpdir()`).

### 2. 락 관리

**File**: `packages/launcher/src/lock.ts` (new)

- `acquire(lockPath)` — `fs.open(lockPath, 'wx')`로 원자적 취득. 성공 시 핸들
  반환, EEXIST 시 `null`.
- `readLock(lockPath)` / `writeLock(lockPath, record)` — `LockRecord`(phase
  판별 유니언) JSON 직렬화/역직렬화. 취득 직후 **즉시** `phase: 'starting'`
  레코드를 쓰고(`owner`=런처 pid, `pid: null`, port는 확정 후 갱신), 서버가
  헬스로 확인되면 `phase: 'running'`으로 전이하며 **`pid`는 `ViewerInfo.pid`
  (Vite 프로세스 자신의 pid)로 채운다**. `pnpm` 래퍼 pid(`child.pid`)가 아니다
  — 런타임 플러그인의 소유권 정리(`lock.pid === process.pid`)가 Vite 프로세스와
  일치해야 하기 때문(새 P1). 빈 락이 절대 남지 않는다(P1-A).
- `isStale(record)` — `starting`이면 `owner`(런처 pid), `running`이면 `pid`
  (Vite pid)를 `process.kill(pid, 0)`로 생존 확인. 죽었으면 stale. 런처는
  기동 성공 후 종료하므로 `running` 판정은 반드시 Vite pid로 한다(owner는 이미
  사라짐).
- **원자적 stale 복구** (P1-B) — stale 판정→제거→재취득을 별도 복구 락
  (`<lockPath>.recover`, `wx`)으로 직렬화한다. 복구 락을 잡은 런처만 stale
  메인 락을 unlink하고 재취득하며, 나머지는 복구 완료를 폴링 후 재사용한다.
  단순 unlink+재취득의 TOCTOU(두 런처가 서로의 새 락을 지움)를 차단한다.

락 파일은 서버 수명 동안 유지된다. **정상 종료 시 삭제 책임은 뷰어 런타임
플러그인(§5)의 종료 훅**에 있고, **startup 실패 시(spawn 에러·포트 소진·헬스
타임아웃) 삭제 책임은 런처(§6)의 실패 경로**에 있다(자신이 소유한 락과 spawn한
자식을 정리). 런처는 startup 판정에서 락을 읽고 stale 시 위 복구 절차로 정리한다.

### 3. 포트 탐색

**File**: `packages/launcher/src/port.ts` (new)

- `resolveBandPort(startPort, docsDir)` — 후보 포트부터 대역 `[PORT_START,
  PORT_END]`를 훑으며 각 포트를 판정한다: bind 가능하면 `{kind:'free', port}`,
  점유돼 있으면 `/api/viewer-info` 헬스로 정체 확인 → 우리 뷰어(docsDir 일치)면
  `{kind:'adopt', port, info}`, 외부 프로세스면 +1. 대역 소진 시 에러 throw.
  락 없이 standalone(`pnpm dev`)으로 이미 떠 있는 동일 뷰어를 중복 기동하지
  않고 채택하기 위해 헬스 판별을 필수로 둔다.

### 4. 헬스체크 클라이언트

**File**: `packages/launcher/src/health.ts` (new)

- `fetchViewerInfo(port, timeoutMs)` — `http://127.0.0.1:<port>/api/viewer-info`를
  전역 `fetch`(Node 22)로 조회. 타임아웃·연결거부는 `null`로 흡수.
- `ViewerInfo` 타입 — `{ name, version, docsDir, pid, startedAt }`.
- `isReusable(info, expectedDocsDir)` — `name === 'previs-viewer'` 그리고
  `docsDir === expectedDocsDir`일 때만 재사용 가능.

### 5. 뷰어 런타임 플러그인 (정체·유휴·락 정리)

**File**: `apps/viewer/server/previs-runtime-plugin.ts` (new)

Vite dev 서버 프로세스 안에서 동작하는 수명주기 플러그인. 기존
`previs-docs-plugin.ts`(문서 API)와 관심사를 분리한다.

- **`/api/viewer-info`** (GET) — `{ name: 'previs-viewer', version, docsDir,
  pid, startedAt }` 반환. version은 `apps/viewer/package.json`에서 읽고,
  docsDir은 런처가 넘긴 canonical docsDir(§1)과 동일 값이어야 재사용 비교가
  성립한다.
- **활동 스탬프** — 모든 요청에서 `lastActivity`를 갱신하되, **반드시
  `previsDocsPlugin`보다 먼저** 실행되게 등록한다(P2-E). 기존 docs
  미들웨어는 `/api/documents`를 `next()` 없이 종료하므로, 뒤에 두면 뷰어의
  포커스 갱신(`/api/documents`)이 idle 타이머를 갱신하지 못해 활성 사용
  중에도 유휴 종료될 수 있다. Vite `enforce: 'pre'` + 플러그인 배열에서 docs
  플러그인 앞에 두는 이중 보장으로 순서를 명시한다.
- **유휴 종료** — 주기 타이머로 `now - lastActivity > 30분`이면 종료.
- **락 정리** (P2-C) — `PREVIS_LOCK_PATH`가 있으면 종료 시 락을 지운다. `exit`
  핸들러는 async를 await할 수 없으므로 **동기 `fs.unlinkSync`**로, **멱등**하게
  (이미 없으면 무시), **소유권 확인**(`running` 레코드의 `pid === process.pid`
  일 때만 — §7에서 Vite 자신의 pid를 기록하므로 일치) 삭제한다. 다른 소유자의
  락은 지우지 않는다.
  `SIGINT`/`SIGTERM` 리스너는 Node 기본 종료를 무력화하므로, 동기 정리 후
  **명시적으로 `process.exit`하거나 시그널을 재-raise**해 프로세스가 실제로
  종료되게 한다. 환경변수가 없으면(standalone `pnpm dev`) 락·유휴 종료를
  비활성화한다(개발 편의 유지).

**Current state**: 파일 없음. 정체·유휴·락 정리 기능이 전무.

### 6. 런처 오케스트레이션

**File**: `packages/launcher/src/launch.ts` (new)

재사용 우선 흐름:

1. `resolveTarget`으로 docsDir·targetKey·lockPath·후보 포트 확정.
2. **재사용 검사** — 락이 있으면 기록 포트로 `fetchViewerInfo` → `isReusable`면
   URL 반환하고 종료. 기록이 `phase: 'starting'`이고 소유 pid가 살아 있으면
   다른 런처가 기동 중이므로 헬스 폴링으로 넘어간다.
3. **락 취득** — `acquire(lockPath)` 성공 시 **즉시 `phase: 'starting'` 레코드
   기록**(owner=자기 pid). 빈 락 구간을 없앤다(P1-A).
   - EEXIST면 다른 런처가 기동 중 → 헬스가 뜰 때까지 바운드 폴링 후 재사용.
     기록이 stale(§2 소유 pid 사망)이면 **복구 락으로 직렬화**된 stale 정리→
     재취득을 시도(바운드)(P1-B).
4. **포트 확정** — `resolveBandPort`. 결과가 `adopt`(락 없이 떠 있는 동일
   뷰어를 후보/대역에서 발견)면 그 pid·port로 `running` 락을 기록하고
   `reused`로 반환한다(중복 기동 방지). `free`면 확정 포트를 락 레코드에 갱신
   기록해 경쟁 런처가 폴링할 수 있게 한다.
5. **spawn** — `spawn('pnpm', ['--filter','@previs/viewer','dev'], {
   detached: true, stdio: ['ignore', logfd, logfd],
   env: { ...process.env, PREVIS_DOCS_DIR, PREVIS_PORT, PREVIS_LOCK_PATH } })`
   후 `child.unref()`. `detached: true`로 자식이 프로세스 그룹 리더가 되어
   Vite 손자까지 그룹 단위로 제어 가능해진다. `pnpm` 래퍼 pid는 실패 정리용
   으로만 보관한다(락에 기록하는 pid와 구분).
6. **헬스 대기** — `/api/viewer-info`가 뜰 때까지 바운드 폴링.
7. **락 확정** — 성공 시 `phase: 'running'`으로 전이하고 `pid`에
   **`ViewerInfo.pid`(Vite 자신의 pid)**를 기록한다(새 P1). `pnpm` pid가 아니다.
8. 뷰어 URL을 stdout에 한 줄로 출력하고 종료(서버는 계속 상주).
9. **실패 경로 정리** (P1-A, 새 P1) — spawn 에러·포트 소진·헬스 타임아웃 등
   어느 단계에서 실패하든: **detached 프로세스 그룹 전체를 종료**
   (`process.kill(-child.pid, 'SIGTERM')`)하고 exit을 바운드 대기, 미종료 시
   SIGKILL 에스컬레이션해 **Vite 손자까지 확실히 죽인 뒤에** 자신이 소유한 락을
   삭제하고 에러로 종료한다. Vite가 포트를 쥔 채 락만 지워져 다음 호출이 중복
   기동하는 상황을 막는다. 소유하지 않은 락은 건드리지 않는다.

**가정**: 이번 사이클 런처는 previs 리포 루트에서 실행된다(cwd = 리포 루트).
외부 프로젝트에서 뷰어 소스 위치를 찾는 문제는 M7 패키징 과제다.

## Technical Considerations

- **Pattern Usage (ARCHI.md)**: 새 패키지는 `@previs/schema`·`@previs/recap`의
  plain `tsc` CLI 패턴을 따른다(§6 Build System). 뷰어 서버 확장은 기존 Vite
  플러그인 미들웨어 패턴(§8)을 따른다.
- **TS 전용 규칙**: 모든 신규 소스는 `.ts` (AGENTS.md). 런처 CLI도 `.ts` →
  `tsc` → `dist/*.js`를 `node`로 실행.
- **락 위치**: `os.tmpdir()` 사용으로 리포 git 오염 방지. `.previs/` 안에 두지
  않는다.
- **detached 프로세스**: macOS에서 `detached: true` + `unref()` + stdio를 로그
  파일로 리다이렉트해 부모(스킬 셸) 종료 후에도 서버가 상주하게 한다.
- **strictPort 정책**: 런처 경로는 `PREVIS_PORT` + `strictPort: true`로 대역
  이탈을 원천 차단. standalone `pnpm dev`는 기존 편의(`false`) 유지.
- **Edge Cases**:
  - 락은 있으나 서버 죽음(비정상 종료로 락 미정리) → 소유 pid 생존 확인 후
    복구 락으로 직렬화된 stale 정리(P1-B).
  - startup 중단(spawn 에러·포트 소진·헬스 타임아웃) → `phase: 'starting'` 락 +
    자식 프로세스를 런처 실패 경로가 정리(P1-A). 빈 락은 존재하지 않는다.
  - 후보 포트를 외부 프로세스가 점유 → 대역 내 +1, 소진 시 명확한 에러.
  - 동시 두 런처 → `wx` EEXIST로 한쪽만 기동, 다른 쪽은 락의 확정 포트가
    기록되면 헬스 폴링 후 재사용.
  - 동시 두 런처가 같은 stale 락 관측 → 복구 락으로 한쪽만 정리·재취득(P1-B).
  - 다른 타깃(docsDir)의 우리 뷰어가 같은 포트에 있음 → `isReusable`가
    docsDir 불일치로 거부, +1 탐색.
  - `PREVIS_LOCK_PATH` 미설정(standalone) → 유휴 종료·락 정리 비활성.
  - 유휴 종료/시그널 → 동기·멱등·소유권 확인 unlink 후 명시적 종료(P2-C).

## Type Definitions

- `ViewerInfo` — `{ name: string; version: string; docsDir: string; pid: number; startedAt: number }` (health.ts / 런타임 플러그인 응답 공유 형태)
- `LockRecord` — phase 판별 유니언 (lock.ts):
  - `{ phase: 'starting'; owner: number; pid: null; port: number | null; startedAt: number }` — spawn 전, `pid` 없음
  - `{ phase: 'running'; owner: number; pid: number; port: number; startedAt: number }` — `pid`=Vite 프로세스 자신의 pid(`ViewerInfo.pid`, `pnpm` 래퍼 아님)
  - `owner`=락을 만든 런처 pid
- `LaunchResult` — `{ url: string; port: number; reused: boolean }` (launch.ts 반환)
- `ResolvedTarget` — `{ targetKey: string; docsDir: string; lockPath: string; logPath: string; candidatePort: number }` — `targetKey === docsDir`(canonical 절대경로) (paths.ts)

## Performance & Cost Impact

- 재사용 경로는 락 읽기 + 헬스 GET 1회(수 ms)로 종료 — 세션당 서버 재기동
  비용 제거가 핵심 이득.
- 최초 기동만 Vite dev 서버 startup 비용(수 초). 이후 세션은 재사용.
- 유휴 30분 종료로 좀비 프로세스·메모리 상주 방지.

## Backward Compatibility

- `pnpm --filter @previs/viewer dev` standalone 실행은 그대로 동작(락·유휴
  비활성, 폴백 포트). 파생 기준을 리포 루트에서 **canonical docsDir**로
  통일(P2-D)하므로 dogfood 포트 값이 현행 vite.config 산출값과 달라질 수 있으나,
  런처 경로와 standalone 경로가 항상 같은 포트를 산출해 서로 다른 서버가 뜨는
  일이 없다(무해한 값 변경, 정체 일관성이 이득).
- `/api/documents` 계약 불변. `/api/viewer-info`는 순수 추가(additive).
- 스킬 인터페이스 변화는 "안내 명령"뿐이라 발행 파이프라인·문서 포맷에 영향 없음.
