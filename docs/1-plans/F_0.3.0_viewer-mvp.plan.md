# 뷰어 MVP — 읽기 전용 렌더링 (M2) Implementation Plan

> 버전: 0.3.0 (M2 완료 시 minor 상승 — GOAL.md TRIP 운영 규칙)
> 범위: GOAL.md M2 전체 (사용자 확정: 단일 TRIP 사이클, 문서 로딩은
> "픽스처 내장 + 파일 열기" 방식)

## Overview

로컬 JSON 블록 문서를 열어 보는 최소 뷰어 `apps/viewer`를 구축한다.
React SPA에 DESIGN.md 토큰을 Tailwind CSS 변수로 매핑(다크 변형 자체 정의,
한글 폰트 폴백)하고, 기본 블록 렌더러 6종(prose, callout, file-tree, tabs,
columns, diff)과 문서 목록(그라디언트 카드 정체성)·문서 뷰를 구현한다.
완료 기준: M1 픽스처(sample-plan/sample-recap)가 라이트/다크 양쪽에서
렌더링된다.

## Problem Statement

- M1에서 스키마 계약은 고정됐지만 문서를 "볼" 수단이 없다. previs의 가치
  (계획·회고의 시각 검토)는 뷰어 없이는 검증 불가능하다.
- ARCHI.md §3의 뷰어·UI 프리미티브 행이 `예정` 상태이고, §8(Components)도
  코드 없이 설계만 존재한다.

## Solution Architecture

```
apps/viewer/                  # @previs/viewer — React SPA (Vite)
├── package.json              # dev/build/typecheck/test 스크립트
├── tsconfig.json             # bundler 모듈 해석, react-jsx
├── vite.config.ts            # react 플러그인 + @tailwindcss/vite + 전용 대역 포트
├── vitest.config.ts          # jsdom 환경, 콜로케이션 테스트
├── components.json           # shadcn/ui CLI 설정
├── index.html
└── src/
    ├── main.tsx              # 엔트리 (폰트 import, 라우터 마운트)
    ├── App.tsx               # react-router 라우트: / (목록), /doc/:id (뷰)
    ├── index.css             # Tailwind 4 @theme — DESIGN.md 토큰 → CSS 변수 + 다크 변형
    ├── lib/
    │   ├── utils.ts          # cn (shadcn 표준)
    │   ├── documents.ts      # 문서 저장소: 내장 픽스처 로딩 + 열린 파일 상태
    │   └── card-identity.ts  # 문서 id → 그라디언트 4색 결정적 매핑
    ├── components/
    │   ├── ui/               # shadcn/ui 프리미티브 (button, tabs)
    │   ├── app/              # 앱 크롬 (DESIGN.md 계층)
    │   │   ├── DocumentList.tsx / DocumentCard.tsx
    │   │   ├── DocumentView.tsx
    │   │   ├── ThemeToggle.tsx
    │   │   └── OpenFileButton.tsx (+ 드래그앤드롭 존)
    │   └── blocks/           # 블록 렌더러 (콘텐츠 계층)
    │       ├── BlockRenderer.tsx      # type → 컴포넌트 매핑 + fallback
    │       ├── ProseBlock.tsx / CalloutBlock.tsx / FileTreeBlock.tsx
    │       ├── TabsBlock.tsx / ColumnsBlock.tsx / DiffBlock.tsx
    │       └── FallbackBlock.tsx      # M3 블록 6종 placeholder
    └── (테스트는 소스 옆 콜로케이션)
```

설계 결정:

1. **Vite 8 + React 19** — vitest 4의 peer 범위(`vite ^6||^7||^8`) 안에서
   최신 조합. 루트 vitest `projects`에 `apps/*`를 추가해 뷰어는 jsdom,
   schema는 node 환경으로 분리 실행한다.
2. **Tailwind CSS 4 (CSS-first)** — `@theme`으로 DESIGN.md 토큰(색·타이포·
   radius·spacing)을 CSS 변수로 매핑. DESIGN.md는 라이트 전용이므로
   (Known Gaps) semantic 토큰 층(`--color-canvas`, `--color-ink` 등)을 두고
   `.dark` variant에서 값을 반전 정의한다 — previs 적용 지침 준수.
3. **폰트 자체 호스팅** — `@fontsource-variable/dm-sans` + `pretendard`
   (variable) npm 패키지. 폰트 스택 `"DM Sans", "Pretendard Variable",
   sans-serif` (DESIGN.md 한글 지침). CDN 미사용(데이터 주권).
4. **시각 계층 분리 준수** — 앱 크롬(목록·카드·내비·버튼)만 DESIGN.md 토큰.
   블록 렌더러는 콘텐츠 계층으로 분리하고 중립 스타일(문서 타이포그래피)로
   구현한다. `--wf-*` 손그림 체계는 M3 범위이며 M2 블록 6종은 손그림 미학
   대상이 아닌 텍스트·구조 블록이다.
5. **문서 로딩 이중 경로** — ① 내장: `@previs/schema` 픽스처 3종을 번들에
   포함해 목록에 표시(kitchen-sink는 fallback 렌더러 시연 겸용). 픽스처는
   schema 패키지에 `./fixtures/*.json` 서브패스 export를 추가(additive)해
   import하며, 뷰어 tsconfig에 `resolveJsonModule`을 켠다.
   ② 파일 열기: 파일 선택 버튼 + 드래그앤드롭으로 임의 JSON을
   `safeParsePrevisDocument`로 검증 후 메모리에 추가. 실패 시 스키마 에러를
   인라인으로 표시(브라우저 alert 금지 — AGENTS.md).
6. **diff 렌더링은 shiki `diff` 문법** — MVP는 unified diff 텍스트를 shiki의
   `diff` 언어로 하이라이트(+/− 행 배경). `lang` 필드 기반 이중 하이라이트는
   M3 diff 심화로 미룬다. shiki 하이라이터는 싱글턴으로 lazy 생성.
7. **알 수 없는/미구현 블록은 FallbackBlock** — 스키마상 유효하지만 M2
   렌더러가 없는 6종(wireframe 등)은 타입명과 "M3에서 지원" placeholder로
   렌더링해 문서 전체 렌더링이 깨지지 않게 한다.
8. **라우팅은 react-router 8** — `/`(목록), `/doc/:id`(뷰). 열린 파일 문서는
   메모리 저장소(React 상태)로 관리 — MVP에서 영속화하지 않는다.
9. **테마 전환** — `prefers-color-scheme` 기본 + 토글(localStorage 유지),
   `html.dark` 클래스 방식. 완료 기준의 라이트/다크 검증 대상.
10. **schema는 소스로 소비** — 뷰어의 `@previs/schema` 해석을 tsconfig
    `paths` + vite `resolve.alias`로 schema `src/index.ts`에 연결한다.
    alias는 **exact-match**(`find: /^@previs\/schema$/` 정규식)로 지정해
    `@previs/schema/fixtures/*.json` 서브패스가 prefix 치환으로 깨지지
    않게 한다(픽스처는 exports 맵으로 해석). clean checkout에서 dist 없이
    dev/typecheck/test가 동작하고 stale dist 문제가 없다. `dist/` exports
    계약은 Node 소비자(M4 스킬·M5 런처)용으로 유지되며 schema의 dist
    smoke test가 계속 보호한다.

## Implementation Details

### 1. 워크스페이스 통합 (루트)

**Files**: `vitest.config.ts`, `package.json` (루트)

**Current state**: vitest `projects: ['packages/*']`, `test` 스크립트가
`pnpm build && vitest run` (전체 재귀 빌드 후 테스트).

**Modifications**:

- vitest `projects`에 `apps/*` 추가.
- 루트 `test` 스크립트의 선행 빌드를 `pnpm --filter @previs/schema build`로
  좁힘 — dist smoke test에 필요한 것은 schema 빌드뿐이며, 뷰어 vite 빌드가
  테스트마다 도는 것을 방지.

### 2. 뷰어 패키지 골격

**Files**: `apps/viewer/package.json`, `tsconfig.json`, `vite.config.ts`,
`vitest.config.ts`, `index.html`, `components.json`, `src/main.tsx`, `src/App.tsx`

- 패키지명 `@previs/viewer`, `private: true`. 의존성: react, react-dom,
  react-router, `@previs/schema`(workspace:*), shiki, react-markdown, 폰트 2종,
  shadcn 런타임(clsx, tailwind-merge, class-variance-authority, radix-ui tabs),
  @tabler/icons-react. devDependencies: vite, @vitejs/plugin-react,
  tailwindcss, @tailwindcss/vite, jsdom, @testing-library/react,
  @testing-library/jest-dom, @types/node, **@types/react, @types/react-dom**
  (JSX typecheck에 필수 — 루트에 없음).
- 스크립트: `dev`(vite), `build`(tsc --noEmit 후 vite build), `typecheck`,
  `test`(vitest run). 루트 `pnpm -r typecheck/build`에 자동 포함.
  설계 결정 10(소스 alias) 덕에 어느 스크립트도 schema dist를 전제하지
  않는다.
- **dev 서버 포트**: AGENTS.md 로컬 뷰어 규칙에 따라 previs 전용 대역
  `47738~47801`에서 프로젝트 경로 해시로 유도한 포트를 `vite.config.ts`에서
  계산해 지정 (3000/5173 등 기본 포트 금지). 유도 로직은 M5 런처가 재사용할
  수 있도록 단순 해시 함수로 config 내에 구현.
- tsconfig: `tsconfig.base.json` extends + `moduleResolution: bundler`,
  `jsx: react-jsx`, vite/client 타입, `resolveJsonModule`(픽스처 import),
  `paths`로 `@previs/schema` → `../../packages/schema/src/index.ts` 연결
  (tsconfig paths는 exact 지정이므로 서브패스 영향 없음).
- vite.config.ts의 `resolve.alias`는 exact-match 정규식
  (`/^@previs\/schema$/`)으로 동일 대상을 지정 — 설계 결정 10 참조.
- `vitest.config.ts`는 vite 설정을 자동 병합하지 않으므로
  `mergeConfig(viteConfig, defineConfig({ test: ... }))` 패턴으로
  vite.config.ts를 명시적으로 병합해 alias·플러그인을 공유한다.

### 3. 디자인 토큰 매핑 (앱 크롬 계층)

**File**: `apps/viewer/src/index.css`

- Tailwind 4 `@theme`에 DESIGN.md 토큰 매핑:
  - 색: canvas/surface/surface-soft/hairline(+soft)/ink/charcoal/slate/steel/
    stone/muted/primary + brand 4색(coral/magenta/blue/purple — 카드 전용).
  - 타이포: DM Sans 스택 + 사이즈 스케일(heading-lg~micro 중 뷰어가 쓰는
    범위), radius 스케일(xs~hero, full), spacing은 Tailwind 기본 4px 체계와
    호환되므로 별도 토큰 없이 표준 유틸 사용.
- semantic 레이어: 라이트 값은 DESIGN.md 그대로, `.dark`에서 canvas/ink 계열
  반전 값을 자체 정의(DESIGN.md Known Gaps — 다크 팔레트 미발행). 대비는
  본문 텍스트 기준 WCAG AA를 목표로 한다.
- 버튼·배지·카드 radius는 DESIGN.md 규칙(버튼 full, 카드 xl, 그라디언트
  카드 hero=32px) 준수.

### 4. shadcn/ui 프리미티브

**Files**: `apps/viewer/components.json`, `src/components/ui/*`, `src/lib/utils.ts`

- shadcn CLI로 `button`, `tabs`만 도입(MVP 최소). 스타일 변수는 3번의
  semantic 토큰에 연결.
- 아이콘은 Tabler Icons로 통일 — shadcn 기본 예제의 lucide 참조는 사용하지
  않는다 (AGENTS.md: 아이콘은 Tabler).

### 5. 문서 저장소와 파일 열기

**Files**: `apps/viewer/src/lib/documents.ts`, `src/components/app/OpenFileButton.tsx`

- 내장 픽스처: `@previs/schema/fixtures/*.json` 3종을 정적 import하고
  `safeParsePrevisDocument`로 파싱해 목록의 초기 데이터로 사용 — 뷰어도
  소비자로서 항상 스키마 검증을 거친다(신뢰 경계 일관성). 이를 위해
  `packages/schema/package.json` exports에 `"./fixtures/*.json":
  "./fixtures/*.json"` 서브패스를 추가한다(additive, 빌드 불필요한 원본
  JSON이라 dist 의존 없음).
- 파일 열기: `<input type="file">` 버튼 + 문서 목록 영역 드래그앤드롭.
  JSON 파싱 → 스키마 검증 → 성공 시 메모리 저장소에 추가하고 해당 문서로
  이동, 실패 시 zod 이슈 요약을 인라인 콜아웃으로 표시.
- id 충돌(같은 id 재열기)은 기존 항목 교체로 처리.

### 6. 앱 크롬 — 문서 목록·문서 뷰·테마

**Files**: `src/components/app/DocumentList.tsx`, `DocumentCard.tsx`,
`DocumentView.tsx`, `ThemeToggle.tsx`, `src/lib/card-identity.ts`

- **DocumentCard**: DESIGN.md 그라디언트 카드(32px radius, brand 4색)를
  문서 식별색으로 전용. `card-identity.ts`가 문서 id 해시로 coral/magenta/
  blue/purple 중 하나를 결정적으로 배정. kind(plan/recap) 배지, 제목
  (card-title 타이포), createdAt·블록 수 메타 표시.
- **DocumentList**: 반응형 카드 그리드(4→2→1열, DESIGN.md 반응형 규칙),
  상단 바(제목 + OpenFileButton + ThemeToggle).
- **DocumentView**: 문서 헤더(제목, kind 배지, source 메타) + 블록 순차
  렌더링. 본문 폭 ~720px(docs-prose-block 규칙). 목록 복귀 내비게이션.
- **ThemeToggle**: 시스템 기본 → 수동 토글, localStorage 유지,
  `html.dark` 클래스 적용. Tabler 아이콘 사용.

### 7. 블록 렌더러 6종 + fallback (콘텐츠 계층)

**Files**: `src/components/blocks/*`

- **BlockRenderer**: `block.type` → 컴포넌트 매핑 테이블. 미등록 타입은
  FallbackBlock. 컨테이너 재귀는 BlockRenderer 재사용.
- **ProseBlock / CalloutBlock**: react-markdown으로 markdown 렌더링(raw HTML
  비활성 기본값 유지 — 콘텐츠 계층에서 임의 HTML 실행 경로 차단, ARCHI §15).
  Callout은 variant 4종(info/warning/danger/success)별 색·Tabler 아이콘.
- **FileTreeBlock**: 플랫 `entries[].path`를 트리로 변환해 들여쓰기 렌더링.
  status별 색(added/modified/deleted/renamed), `from` 원경로·`note`·
  `inferred` 배지 표시.
- **TabsBlock**: shadcn Tabs 기반, item당 `blocks[]`를 BlockRenderer로 재귀.
- **ColumnsBlock**: CSS grid(2~4열, 모바일 1열 스택), 열마다 재귀 렌더링.
- **DiffBlock**: 헤더(title, file 경로, note) + shiki `diff` 하이라이트
  본문. 하이라이터 싱글턴 lazy 초기화, 로딩 전 plain 텍스트 표시(레이아웃
  시프트 최소화). 라이트/다크 테마 쌍 사용.

### 8. 문서 갱신

**Files**: `docs/ARCHI.md`, `docs/4-unit-tests/TESTING.md`

- ARCHI.md: §3 뷰어·UI 프리미티브·콘텐츠 렌더링(shiki) 행 확정 전환,
  §4 구조에 apps/viewer 반영, §8 `(예정)` 해제(구현 상태 기준으로 개서),
  §14에 뷰어 jsdom 프로젝트 추가.
- TESTING.md: 뷰어 테스트 환경(jsdom·Testing Library) 추가.

## Technical Considerations

- **Pattern Usage**: ARCHI §5 원칙 — 블록 렌더러 매핑(§8), 시각 계층 분리
  (크롬=DESIGN.md / 블록=콘텐츠 계층), JSON 블록만 렌더링(MDX·임의 코드 없음).
- **블록 스키마 영향**: 없음 — 스키마는 소비만 한다. 뷰어는 항상
  `safeParsePrevisDocument`를 거친 문서만 렌더링.
- **Grounding**: 해당 없음 (렌더링만, 문서 생성 없음).
- **시각 계층**: 이 계획의 핵심 경계. 앱 크롬(목록·카드·버튼·배지)은
  DESIGN.md 토큰, 블록 콘텐츠는 중립 문서 타이포. `--wf-*`·rough.js는 M3.
  그라디언트 brand 4색은 문서 카드 식별색으로만 사용(DESIGN.md Don't 준수).
- **Supabase 영향**: 없음 (로컬 전용 MVP).
- **성능**: 블록 컴포넌트 memo화(문서는 불변 데이터), shiki 하이라이터
  싱글턴 + 코드 스플리팅(diff 블록 lazy). 대형 diff 가상화는 M3 이후
  (픽스처 규모에선 불필요). 픽스처 정적 import는 번들 크기에 무해한 수준.
- **Edge Cases**: 스키마 검증 실패 JSON 열기(인라인 에러), 알 수 없는 블록
  타입(스키마가 거부 — 도달 불가, 미구현 타입은 fallback), 빈 목록(내장
  픽스처로 항상 비어있지 않음), 문서 id 충돌(교체), 잘못된 라우트 id(목록
  으로 안내), shiki 초기화 전 렌더(plain 폴백).
