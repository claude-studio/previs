# /previs-recap — diff 기계 도출 recap 파이프라인 구현 계획 (M4 사이클 B)

## Overview

M4(에이전트 스킬)의 마지막 사이클로, 구현 완료 후 git diff에서 recap 문서를
도출하는 `/previs-recap` 스킬을 구현한다. Grounding·Budget·시크릿 마스킹을
모델 재량에 맡기지 않도록, 구조화 블록(file-tree·diff)은 TypeScript CLI
(`@previs/recap`)가 git diff에서 기계적으로 생성하고, 서사·와이어프레임·
리스크 판단은 에이전트가 작성하는 **하이브리드** 구조를 취한다.
완료 시 M4가 끝나고 버전은 v0.5.0(minor)이 된다.

## Problem Statement

- plan 발행 파이프라인(v0.4.1)은 있지만 recap 경로가 없다. 리뷰어가 raw
  diff 이전에 변경의 "형태"를 파악하는 GOAL §검토 품질의 절반이 미구현.
- 콘텐츠 규칙(Grounding·Budget·마스킹·Skip)을 스킬 지침만으로 강제하면
  전사 오류·마스킹 누락이 모델 재량에 달린다. ARCHI §14는 "diff→블록
  도출 로직"을 스키마 검증 다음의 테스트 우선순위로 이미 명시하고 있다
  — 코드로 구현하고 테스트한다는 전제.

## Solution Architecture

```
/previs-recap 스킬 (skills/previs-recap/SKILL.md)
    │ ① diff 소스 확정 (auto | --range | --staged | --worktree)
    ▼
pnpm recap:derive [옵션] --out <manifest.json>     ← 기계 도출 CLI
    │  packages/recap: git diff → 인벤토리·발췌·마스킹·skip 판정
    │  산출: RecapManifest { fileTree 블록, diff 블록 후보(랭킹), stats, skip }
    ▼
스킬: ② skip 권고면 중단 보고
     ③ recap 표준 골격 조립 — wireframe 헤드라인 → prose 서사(1~3문단)
        → data-model/api-endpoint(해당 시) → file-tree(CLI 산출 그대로)
        → tabs "Key changes" (CLI diff 후보 중 최대 8개 선택)
    ▼
.previs/recap-YYYYMMDD-<slug>.json → pnpm doc:validate 루프 → 뷰어 확인
```

- **역할 분담이 Grounding 경계**: 구조화 블록의 *내용*은 CLI 산출물을 변형
  없이 사용한다. 에이전트의 판단은 (a) key-change 탭 *선택*, (b) 산문·
  와이어프레임 작성, (c) diff에 실재하는 스키마/API 변경의 data-model·
  api-endpoint 전사(불확실하면 `inferred`)에만 허용된다.
- **마스킹은 에이전트 손에 닿기 전에**: CLI가 diff 텍스트를 출력하기 전에
  패턴 기반 마스킹을 적용한다. 스킬 지침의 마스킹 규칙은 이중 방어.
- **기존 발행 규약 재사용**: `.previs/` 저장, 파일명=문서 id, 접미사
  유일화, `doc:validate` 게이트, 뷰어 focus refetch — v0.4.1 파이프라인
  그대로. 이번 사이클도 발행 대상은 previs 리포 도그푸딩으로 한정한다.

### 2. diff 소스 해석

**File**: `packages/recap/src/source.ts` (new)

결정적 해석 규칙 (사용자 확정: 자동 감지 + 인자 오버라이드):

1. `--range <rev-range>` 명시 시 그대로 사용 (예: `v0.4.0..v0.4.1`).
2. `--staged`: index vs HEAD.
3. `--worktree`: working tree vs HEAD (staged 포함).
4. 인자 없음(auto):
   - base 감지: `origin/HEAD` 심볼릭 ref → 실패 시 로컬 `main` → `master`.
   - **브랜치 동치는 논리적으로 판정**: base ref의 short name에서 원격
     접두(`origin/`)를 제거한 브랜치명과 현재 브랜치명을 비교한다
     (`origin/main` vs `main` 문자열 불일치로 오분기하지 않는다).
   - **더티 상태 게이트**: auto 모드에서 워킹트리에 변경(스테이징·
     미스테이징·untracked 포함)이 있으면 조용히 커밋 범위만 다루지 않고
     오류 종료하며 `--range`/`--worktree` 명시 또는 커밋을 안내한다
     — 부분 diff의 조용한 선택 금지.
   - 현재 브랜치 ≠ base(논리 동치 기준) → `merge-base(base, HEAD)..HEAD`.
   - 현재 브랜치 = base → "범위를 명시하라"는 안내와 함께 오류 종료
     (조용한 오추론 금지).
- 해석 결과를 문서 `source` 메타(`branch`, `commitRange`, `mode`)로 반환.
  worktree/staged 모드는 `commitRange` 없이 `branch`·`mode`만 기록.

### 3. git 실행 경계

**File**: `packages/recap/src/git.ts` (new)

- git 서브프로세스 래퍼 단일 모듈. 파싱 로직은 전부 git 출력 *문자열*을
  입력으로 받는 순수 함수로 분리해, 테스트가 git 실행 없이 캡처된 출력

### 4. 파일 인벤토리 → file-tree 블록

**File**: `packages/recap/src/inventory.ts` (new)

- `git diff --name-status -M -z` 파싱: `A/M/D/R<score>` →
  `added/modified/deleted/renamed`(+`from`). `-z` 사용으로 특수 문자
  경로 안전.
- `git diff --numstat -z`로 파일별 churn(추가+삭제 라인) 산출. 바이너리
  (`-` 표기)는 diff 발췌 대상에서 제외하고 file-tree `note`에 표기.
- **untracked 파일 포함 (worktree 모드)**: `git diff`는 untracked를
  다루지 않으므로 `git ls-files --others --exclude-standard`로 수집해
  `added` 엔트리로 인벤토리에 포함하고, 발췌·numstat은
  `git diff --no-index /dev/null <file>` 합성 패치로 산출한다 —
  구현이 새 파일을 추가하는 흔한 경우가 recap에서 조용히 누락되지
  않게 한다. staged 모드는 index 대상이므로 해당 없음, range 모드는
  커밋된 내용만 다루므로 해당 없음(더티 상태는 §2 게이트가 차단).
  **exit code 규약**: `--no-index`는 차이를 찾으면 exit 1을 반환하므로
  합성 패치 실행은 0·1을 성공으로 취급하고 1 초과만 치명 오류로
  분류한다(§7 오류 규칙) — untracked 파일이 있을 때마다 도출이
  중단되는 오분류 방지.
- 산출: 스키마 그대로의 `FileTreeBlock` (전체 파일 — Grounding은 완전한
  인벤토리를 요구, 선별하지 않는다).

### 5. diff 발췌 → diff 블록 후보

**File**: `packages/recap/src/excerpt.ts` (new)

- 파일별 unified diff(context 3)를 발췌해 스키마 그대로의 `DiffBlock` 생성.
- **Budget 강제 (AGENTS.md 콘텐츠 규칙 3)**: 발췌는 150줄 미만. 초과 시
  hunk 경계에서 절단하되, 첫 hunk부터 초과하면 hunk 내부 절단을 허용.
  절단 사실은 `note`에 기계적으로 기록 (예: "전체 diff 412줄 중 148줄
  발췌") — 이것도 grounded 사실이다.
- **마스킹된 전체 diff는 out-of-line**: 파일별 절단 없는 전체 diff는
  마스킹을 적용해 manifest 옆 `<manifest>.patches/` 디렉토리에 patch
  파일로 저장하고, manifest의 각 diff 항목에는 그 경로(`fullDiffPath`)
  만 기록한다. 에이전트가 data-model·api-endpoint 전사 근거로 발췌
  (절단본)만 보거나 마스킹을 우회해 raw `git diff`를 직접 읽는 일이
  없도록 완전한 마스킹 증거를 제공하되, lockfile·생성물 같은 대형
  diff가 manifest를 인라인으로 부풀려 에이전트 컨텍스트를 넘치게
  하지 않게 필요한 파일만 온디맨드로 읽게 한다.
- `title`: `<경로> (+추가 −삭제)` 형식, 70자 초과 시 경로 앞부분 축약.
- `lang`: 확장자 매핑(ts/tsx/json/md/css 등), 미상이면 생략.
- 후보 랭킹: churn 내림차순. lockfile·생성물 패턴(`pnpm-lock.yaml`,
  `dist/` 등)은 최하위로 강등하고 `demoted` 사유를 표기 — 제외가 아니라
  강등이므로 에이전트가 의미 있는 경우 여전히 선택할 수 있다.

### 6. 시크릿 마스킹

**File**: `packages/recap/src/masking.ts` (new)

- CLI가 출력하는 모든 diff 텍스트(발췌 `block.diff`와 out-of-line
  전체 patch 파일 둘 다)에 출력 전 일괄 적용. 패턴(보수적, 과탐지 허용):
  - 알려진 토큰 접두: `sk-`, `ghp_`/`gho_`/`ghs_`, `xox[abps]-`, `AKIA`,
    `ya29.`, JWT(`eyJ` 3분절) → `sk-•••` 형식으로 접두만 남김
  - PEM 블록(`-----BEGIN … PRIVATE KEY-----`) → `<redacted>`
  - `.env*` 파일 diff는 모든 값(`KEY=` 우변)을 `<redacted>`
  - `key/token/secret/password` 계열 이름의 할당 우변
- 원칙: 마스킹 과잉이 원문 노출보다 낫다. 패턴 목록은 테스트 픽스처와
  함께 관리한다.

### 7. Skip 판정 + manifest 조립 + CLI 엔트리

**File**: `packages/recap/src/manifest.ts`, `packages/recap/src/cli/derive.ts` (new)

- **Skip (콘텐츠 규칙 6)**: 변경 파일 1개이고 총 변경 라인 ≤ 40이면
  `skip: { recommended: true, reason }` 반환. `--force`로 무시 가능.
  판정만 하고 종료 코드는 정상 — 최종 중단 결정은 스킬이 사용자에게
  보고하며 내린다.
- manifest 전체를 JSON으로 `--out <path>`에 기록. **`--out`은 필수** —
  out-of-line patch 디렉토리(`<manifest>.patches/`)가 manifest 경로에서
  유도되므로 stdout 모드를 두지 않는다. 누락 시 사용법 안내와 함께
  오류 종료. 구조는 Type Definitions 참조.
- **오류 규칙**: 소스 해석 실패·git 실행 실패 같은 치명 오류는 manifest
  없이 exit ≠ 0으로 종료한다. 파일 단위 비치명 문제만 `errors`에 기록
  하며, `errors`가 비어 있지 않은 manifest는 스킬이 발행하지 않는다
  (§8) — 부분 manifest가 조용히 발행되는 경로를 없앤다.
- CLI 인자: `--range | --staged | --worktree | --force | --out`.
  `validate.ts`와 동일한 엔트리 패턴(`main(args)` + exit code, 테스트
  가능한 export).

**File**: `package.json` (루트, modify)

- `"recap:derive"` 스크립트 추가 — `doc:validate` 선례대로 의존 빌드 후
  `node packages/recap/dist/cli/derive.js` 실행
  (`--filter @previs/recap...`로 schema까지 빌드).

**File**: `packages/schema/src/document.ts` (modify)

**Current state**: `sourceSchema`는 `branch`·`commitRange`·`pr`만 갖는다
(7행) — staged/worktree recap이 발행되면 어떤 상태를 다룬 문서인지

### 8. /previs-recap 스킬

**File**: `skills/previs-recap/SKILL.md` (new)

`previs-plan` 선례를 따르는 절차서. frontmatter:
`name: previs-recap`, `disable-model-invocation: true`(파일 쓰기 워크플로우).

1. **diff 소스 확정**: 사용자 인자가 있으면 CLI 옵션으로 전달, 없으면
   auto. 도출 전에 어떤 범위를 다루는지 사용자에게 확인 가능하게 명시.
2. **도출**: `pnpm recap:derive [옵션] --out <스크래치 경로>` 실행 후
   manifest 로드.
3. **Skip·오류 게이트**: `skip.recommended`면 발행하지 않고 사유를
   보고하고 종료 (사용자가 강행을 지시하면 `--force`로 재도출).
   `errors`가 비어 있지 않아도 발행하지 않고 내용을 보고한다.
4. **recap 표준 골격 조립 (콘텐츠 규칙 2)**:
   - `wireframe` 또는 `diagram` — 변경의 형태를 보여주는 헤드라인
     (`--wf-*` 체계, hex·좌표 하드코딩 금지)
   - `prose` — 1~3문단 서사 (왜·무엇이 바뀌었나 — 자유 서술 허용 영역)
   - `data-model`·`api-endpoint` — diff에 실재하는 스키마/API 변경이
     있을 때만. **전사 근거는 manifest가 가리키는 마스킹된 전체 patch
     파일(`fullDiffPath`)만 사용**하고 raw `git diff`를 직접 실행하지
     않는다(마스킹 경계 유지). diff에서 확인되는 사실만, 불확실한
     항목은 `inferred`
   - `file-tree` — manifest의 블록을 **변형 없이** 사용
   - `tabs` "Key changes" — manifest diff 후보 중 최대 8개 선택(랭킹
     참고, 강등 파일은 의미 있을 때만). **3~8개는 상한 중심의 budget
     이며 후보가 3개 미만이면 있는 만큼만 만들고, 후보가 0개(바이너리
     전용 diff 등)면 tabs를 생략한다** — plan 스킬의 "해당 없는 블록
     생략" 선례. diff 텍스트·title·note 수정 금지, 판단 서술이
     필요하면 별도 prose에
5. **발행·검증**: `kind: recap`, `source`는 manifest 값 사용. id
   `recap-YYYYMMDD-<slug>`, 충돌 시 `-2`·`-3` 접미사, 덮어쓰기 금지.
   `pnpm doc:validate` 실패 시 수정·재검증 루프, 실패 원본 보존.
6. **뷰어 안내**: dev 서버 URL 안내 또는 `pnpm --filter @previs/viewer dev`
   수동 기동 안내 (M5 런처 전 임시).

**File**: `.claude/skills/previs-recap` (new, 심링크 → `../../skills/previs-recap`)

## Technical Considerations

- **Pattern Usage**: CLI 엔트리·빌드·테스트는 `@previs/schema`의
  `validate.ts` 선례를, 스킬 구조·발행 규약은 `previs-plan` 선례를 따른다.
- **Grounding 경계의 코드화**: 구조화 블록 생성이 CLI에 있으므로 "diff에
  없는 사실"이 들어갈 경로가 에이전트 재량에서 제거된다. 에이전트 재량은
  선택·서사·전사(data-model/api-endpoint)로 좁혀지고, 전사 영역만 스킬
  지침으로 통제한다.
- **스키마 변경 최소**: diff·file-tree·tabs 블록은 이미 요구를 충족한다.
  유일한 변경은 `source.mode` optional 필드 추가(§7-1)로 additive 규칙을
  만족하며 기존 문서·픽스처에 영향이 없다.
- **git 파싱 견고성**: `-z` 종결자 사용(경로 공백·한글), rename score
  파싱, 바이너리 numstat `-` 처리. 파싱은 순수 함수 + 캡처 픽스처로
  테스트하고 git 실행은 경계 모듈에 격리한다.
- **Edge Cases**: 빈 diff(오류 안내 종료), 바이너리 파일(발췌 제외·note
  표기, 전부 바이너리면 tabs 생략), rename+수정 동시, 대형 단일 hunk
  (내부 절단), lockfile만 변경된 diff(전부 강등 — 에이전트가 판단),
  `.env` 파일 diff(값 전량 마스킹), merge-base 부재(오류 안내),
  detached HEAD(branch 생략), auto + 더티 워킹트리(오류 안내 종료),
  worktree 모드의 untracked 파일(합성 패치로 포함), 후보 3개 미만
  (있는 만큼만 탭 구성).

## Type Definitions

`packages/recap/src/manifest.ts` (블록 타입은 `@previs/schema` 재사용):

- `DiffSourceResolution`: `{ mode: 'range' | 'staged' | 'worktree';
  branch?: string; commitRange?: string }`
- `RecapManifest`: `{ source: DiffSourceResolution; stats: { files:
  number; insertions: number; deletions: number }; skip: { recommended:
  boolean; reason?: string }; fileTree: FileTreeBlock; diffs: { block:
  DiffBlock; fullDiffPath: string; churn: number; rank: number;
  demoted?: string }[]; errors: string[] }` — `fullDiffPath`는 마스킹
  적용·절단 없는 파일 전체 diff를 담은 out-of-line patch 파일 경로
  (전사 근거용, 온디맨드 로드)
- `@previs/schema`: `DocumentSource`에 `mode?: 'range' | 'staged' |
  'worktree'` 추가 (additive, §7-1). 그 외 공개 API 변경 없음.

## Performance & Cost Impact

- git diff 서브프로세스 수 회 + 문자열 파싱 — 대형 리포에서도 발행
  1회당 수 초 이내 (GOAL §속도). 상주 프로세스·watch 없음.

## Backward Compatibility

- 스키마 변경은 `source.mode` optional 필드 추가뿐 — additive이며
  기존 문서·픽스처는 수정 없이 계속 유효하다. 뷰어 변경 없음. 기존
  `/previs-plan` 파이프라인과 발행 규약을 공유하며 서로 간섭하지
  않는다. 신규 패키지는 additive.
